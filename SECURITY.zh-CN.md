## 报告漏洞

我们认真对待安全问题。如果你发现安全漏洞，请负责任地报告。

### 报告方式

**不要**为安全漏洞创建公开 GitHub Issue。

优先使用本仓库的 GitHub **Private vulnerability reporting** 功能。

如果私密报告不可用，请通过仓库所有者 GitHub 主页关联的联系方式联系维护者，并避免公开发布利用细节。只有在专用安全邮箱已经配置并持续监控后，才应把邮箱地址加入本文件。

### 请包含的信息

1. **描述**：漏洞的清晰说明
2. **影响**：潜在影响和严重程度
3. **复现步骤**：详细复现步骤
4. **受影响版本**：哪些版本受影响
5. **建议修复**：如有可选修复建议，可一并提供

### 响应时间

- **初始响应**：48 小时内
- **状态更新**：7 天内
- **修复时间**：取决于严重程度
  - Critical：24-48 小时
  - High：7 天
  - Medium：30 天
  - Low：下一个版本

### 披露政策

- 我们会确认收到报告
- 我们会提供预估修复时间
- 修复完成后会通知报告者
- 如报告者愿意，会在发布说明中致谢
- 请在修复发布前不要公开披露漏洞细节

## 部署安全建议

部署 MnemoNAS 时请注意：

### 网络安全

1. **使用 HTTPS**：生产环境建议放在带 TLS 的反向代理后
2. **防火墙**：只允许可信网络访问
3. **内部数据面端口**：不要把 dataplane gRPC/HTTP 端口 `9090/9091` 暴露到公网或不可信网络
4. **VPN**：远程访问建议考虑 VPN

### 认证

1. **强密码**：为 Web UI 账号和 WebDAV 认证使用强密码
2. **修改初始凭据**：首次登录后修改初始管理员密码
3. **关闭不用的入口**：不使用 WebDAV 或分享时可关闭对应功能

### 数据保护

1. **备份**：保持定期备份，参见 [备份指南](docs/backup-guide.md)
2. **静态加密**：按部署环境考虑磁盘或文件系统加密
3. **访问控制**：最小化系统权限和网络访问面

### 更新

1. **保持更新**：及时更新 MnemoNAS 和依赖
2. **监控公告**：关注安全公告
3. **先测试**：升级前尽量在测试环境验证

## 依赖检查

发布前和依赖更新后运行漏洞检查：

```bash
make install-audit-tools
make security-check

# 如需包含前端 npm audit，并接受依赖树发送到当前 npm registry：
make security-check NPM_AUDIT=1
```

默认情况下，`make security-check` 使用 `govulncheck` 检查 Go 依赖，并使用 `cargo audit` 检查 dataplane 和 `tools/proto-gen` 的 Rust 依赖。前端 `npm audit --audit-level=high` 需要通过 `NPM_AUDIT=1` 显式启用，因为它会把依赖树发送到当前 npm registry。CI 会在配置的仓库事件中运行 Go、Rust 和前端依赖检查。

## 已知安全注意事项

### 当前限制

1. **无细粒度 ACL**：支持用户、角色和用户根目录，但不支持逐文件 ACL
2. **默认 HTTP**：内置 TLS 可配置，但生产 HTTPS 更建议由反向代理处理
3. **面向本地网络**：不建议在没有加固代理/VPN 的情况下直接暴露到公网

### 计划中的安全能力

- [ ] 更细粒度的分享与访问策略
- [ ] 双因素认证
- [ ] 外部身份提供商集成
