# Development Change Record (Unreleased)

English | [简体中文](release-notes.md)

This document records changes and validation evidence for the current unreleased development branch while retaining preparation steps for a future first public release. It is not a release announcement; MnemoNAS has not published any usable release.

## Summary

The current development branch focuses on stability, public-access safety boundaries, deployment verifiability, and documentation maintainability. Hardening changes are split by risk area. Validation results apply only to the specific target and source tree recorded below; they do not establish that every current unpublished change has been validated, and they do not establish a usable release.

## Major Changes

- Added the `client/` cross-platform Flutter project, with Android as the first usable-platform target. The current source covers server connection and validation, Bearer sessions, single-use refresh-token rotation, secure session storage, the device overview, file browsing and core file operations, version history and administrator restore, recoverable upload and download, Android Storage Access Framework import and export, account management, and an Issue feedback entry. The client gates and a live `nasd` API file-lifecycle smoke test pass. This local manual acceptance pass exercised basic installation, sign-in, system-back hierarchy, session recovery after process restart, and a same-signature Debug update on an Android 16 physical device; it has no corresponding remote device-farm record or downloadable device log. A controlled Release candidate, the complete Android device/failure matrix, background transfers, and native Linux and Windows build validation remain incomplete; no usable client version is available.
- Android release builds now have a fail-closed signing gate. Release retains `com.mnemonas.app`; Debug and Profile use separate package IDs, version suffixes, and display names. A release APK/AAB must load its signing configuration and keystore from outside the source checkout and pass private-key alias, certificate-validity, Android Debug certificate, and certificate SHA-256 fingerprint checks. The gate rejects `android.injected.signing.*` overrides and inspects Gradle's resolved task graph so task abbreviation or exclusion of the standalone validation task cannot bypass the policy. A temporary PKCS12 test key covers missing and invalid input plus both bypass regressions, and the generated APK and AAB undergo actual signature verification. The script removes the test key and release outputs, and CI runs the same gate. This does not establish production-key custody or produce a distributable candidate.
- The Flutter Files page now implements the Android system-back hierarchy: an active selection is cleared first, a child directory returns to its canonical parent, and only the root permits route exit. A directory-list response must also return a canonical path matching the request; an abnormal path is rejected as an invalid response. This local manual acceptance pass separately exercised selection-first back handling, nested-directory back navigation, session recovery after force-stop, and session retention across a Debug `versionCode 1 → 2` update on an API 35 emulator and an OPPO PLP120 running Android 16 (API 36); the temporary environment and test applications were removed afterward.
- Added explicit directory-freshness and file-mutation single-flight boundaries to the Flutter client. Initial-load failure, same-directory refresh, and stale cached content have distinct states, and an optional statistics failure does not mark the file directory stale. A controller lease and immediate presentation-layer debounce keep file mutations single-flight. A confirmed operation is not rewritten as a failure when the directory refresh fails, an unconfirmed result requires inspection before another submission, and a late refresh does not navigate back to an older directory. On entry to `hidden`, `paused`, or `detached`, durable transfer-center tasks retain private payloads and partial progress and become paused or await a newly selected destination; transient file previews, historical-version previews or downloads, and SAF upload preparation are cancelled with partial-file cleanup. Returning to the foreground never resumes a task automatically.
- Added a version-history sheet to the Flutter client. The entry is available to versioned regular files with a canonical path and concrete-read permission and does not require the directory entry to carry a content hash in advance. The version API establishes the current BLAKE3 content identity, and an identity already present on the directory entry must match. The list validates the continuous sequence, hash, size, timestamp, and bounded comment. The current-version card displays identity and metadata only; historical versions can be previewed or downloaded, while current-file actions remain on the Files page. A historical download requires a strong ETag exactly equal to the selected hash and a matching length, and Android opens the system save location only after network completion. Before an administrator restore, the client reloads the parent directory and history, validates the current identity established by the initial version response and the selected entry, and reuses the file-mutation single-flight lease. The restore `POST` is never replayed automatically; connection loss, timeout, unreadable responses, and structured `5xx` failures become unconfirmed results. If restore is confirmed but history refresh fails, the sheet blocks further restore submissions and requires a fresh review after closing.
- Added a durable foreground-transfer ledger to the Flutter client. Tasks are scoped by server and user and record stable private payloads, authoritative offsets, destinations, and phases without storing authentication tokens. A resumed download is appended only when the server object identity, `206 Partial Content`, exact `Content-Range`, and total size all match. Android completes an app-private download payload before selecting a destination with a temporary write grant, so network failure does not create an empty document. Android upload selection uses URI-only `ACTION_OPEN_DOCUMENT` and streams one file at a time under a temporary read grant, avoiding complete multi-selection contents in the Java heap. A known size above 10 GiB is rejected before copying. When metadata omits the size, the native copy layer still enforces a 10 GiB hard limit and stops writing and removes the partial file when the limit is reached. Preparation progress is shown as a separate phase. The temporary import is released after the task-owned payload has been SHA-256 verified and durably recorded.
- Added a dedicated transfer center grouped into active, attention-required, and recent records. Paused or attention-required tasks can be resumed, and completed private downloads can continue to destination selection. Paused or failed tasks require explicit confirmation before any related server session and local recovery progress are removed, while unconfirmed-result records direct the operator to verify the target location first. Task cards also provide file-specific action labels, progress semantics, and narrow-screen, large-text, and desktop-width adaptations.
- Added owner-isolated durable upload sessions with request-ID idempotent creation, read-only lookup by request ID, status lookup, sequential 8 MiB chunks, a server-authoritative offset, chunk SHA-256, complete-payload BLAKE3, conditional commit, cancellation, 72-hour expiry cleanup, and crash-result reconciliation. Non-final chunks are at least 1 MiB. Durable staging defaults to 20 GiB per account and 100 GiB per process, and each write also preserves the configured minimum host free space. The client durably records the attempt before its first creation request. If that response is lost before a session ID is obtained, it only looks up an existing session by client request ID and does not repeat creation. After other lost responses or client restart, the client queries the session before continuing from the server offset. For a successful upload, it deletes the private payload only after observing `committed`; startup retries residual cleanup for confirmed completed or cancelled tasks. If a session is missing or expired, the task becomes unconfirmed instead of recreating a session against a new target snapshot and potentially overwriting a newer file. While the storage recovery gate remains active, the server preserves `committing` and the staged payload instead of confirming publication from an unreconciled visible target. The legacy `POST /api/v1/files/{path}` still accepts only a complete-file request and rejects `Content-Range`.
- The Flutter client session store now uses revision/CAS and atomic `takeAndClear`. A refresh token is removed from durable storage before it is sent. Sign-out clears the local session first and then attempts server-side revocation. Older clients, server endpoints, accounts, and directory responses superseded by a later request can no longer write current state. The controller closes the previous `ApiClient`, isolates server and account state by context epoch, and gives the latest directory request precedence within one context. Deterministic tests cover delayed refresh across sign-out, stale refresh failure concurrent with a new session, delayed sign-in, old-endpoint directory responses, reverse directory completion, and late responses after controller disposal.
- The Flutter client replaces the primary Photos destination, which has no usable index, with Trash. It shows per-item expiry and the current deletion policy, restores to the original or a custom path, and permanently deletes the exact ID set frozen at confirmation time. The client strictly validates Trash IDs, paths, timestamps, counts, and deleted/remaining/skipped partitions. When a request outcome is not confirmed, it reloads Trash but does not infer restore or permanent-delete success only because an item disappeared, and it pauses later mutations until another explicit refresh. Read-only `guest` accounts do not receive mutation controls, and destructive controls expose touch targets of at least 48 dp.
- The Flutter client now provides bounded filename search. Queries are limited to 100 Unicode code points and each request displays at most 100 results. A new query, clear action, search close, sign-out, or server switch cancels the older request, while request sequencing prevents late responses from replacing current state. A failed refresh for the same query retains and marks the previous results. Files are resolved from a freshly loaded parent directory before opening, directories are reloaded before entry, and search snapshots are not used for mutations. The live `nasd` smoke test covers search visibility after upload, rename, move to Trash, restore, and permanent cleanup. The server also corrects byte-based limits for multibyte queries and prevents the search root from appearing as a regular result.
- Streamed uploads, WebDAV PUT, and version restores now use durable write decisions across the namespace, CAS, and SQLite. Startup recovery rolls forward only `committed` transactions, rolls back all others, and retains evidence while blocking later writes when the terminal state cannot be confirmed. The target's direct parent must be created through an explicit directory operation first.
- Strengthened path, archive-download, WebDAV, public-share, workspace, CAS, and backup-restore boundaries, covering symlinks, traversal, percent-encoded dot segments, encoded query or fragment markers, percent-encoded sensitive parameter names, control characters, and rollback error paths.
- Upgraded `golang.org/x/image` to `v0.43.0` to resolve TIFF/WebP dependency-security findings reachable from thumbnail decoding, with the indirect `golang.org/x/text` version refreshed as part of the Go module update.
- Upgraded the Rust data plane and protobuf generator lockfiles to `anyhow 1.0.103`, resolving the `Error::downcast_mut()` memory-safety issue described by `RUSTSEC-2026-0190`.
- Expanded backend and frontend coverage for authentication, users, home directories, directory quotas, directory access rules, share policies, and secure session defaults.
- Added Account Security for every authenticated user, reachable from the user menu, with the administrator Settings page linking to the same entry point. Voluntary and required password changes share validation and error handling, sign the account out on every device after success, and require a new sign-in. When the client can observe an unconfirmed request outcome, it clears local authentication state so the browser does not continue using a potentially revoked session. Requests to `POST /api/v1/auth/password` must now include an `expected_user_id` that matches the authenticated context; existing API callers that omit it receive `400 MISSING_EXPECTED_USER_ID`.
- Single-item and batch deletion on the Files page now share one flow session. Preparation, confirmation, and commit phases are mutually exclusive so separate entry points cannot open or submit concurrent deletions. Delete-target preparation shows cancellable progress. Cancellation immediately aborts the request, ignores late responses, preserves the current selection, and restores focus to the originating control. If that control has been removed, focus falls back to the stable file region. Directory changes and page unmounts invalidate the old session without returning focus to the previous directory.
- The Files page now distinguishes moving an item to Trash from permanent deletion under the current policy, and disables only deletion when that policy is unknown. File lists return an object-identity token for each real item; delete confirmation submits the selected items' observed identities through `POST /api/v1/files-delete-intents` and atomically captures the complete policy and target-tree tokens. Replacing a selected path with a new file or directory invalidates the old identity even when its type, size, and modification time match. Target tokens use a v3 hierarchical SHA-256 Merkle representation that binds the canonical root path, complete path hierarchy, object identities, types, permission modes, sizes, nanosecond modification times, and regular-file content digests. Delete confirmation samples the complete policy and mutation epoch under the storage read lock, scans and hashes the requested target batch outside the lock, then validates the epoch again and retries when it changed. The final target token remains an opaque 64-character lowercase hexadecimal string; the mutation epoch is not exposed through REST. `DELETE /api/v1/files/{path}` requires the mode, policy token, and target token. A policy or target change detected before atomic capture returns `409 DELETE_POLICY_CHANGED` or `409 DELETE_TARGET_CHANGED` without committing business state. After capture, both delete modes use a source-local no-replace stage and verified content evidence. Permanent deletion uses a handle-anchored quarantine with permissions no broader than mode `0700`; incomplete cleanup after logical commit returns success with `delete cleanup incomplete`. Live delete-to-Trash and restore-from-Trash use `prepared`, `copying`, `ready`, `committed`, and `completed` checkpoints under `.transactions`, an SQLite `trash_operations` outbox row committed atomically with business metadata, and operation-scoped share and favorite receipts plus original delete ownership. The `copying` checkpoint durably identifies a synced operation-owned container before payload copying. Before that checkpoint, a canonical mode-`0600` owner marker binds the `prepared` journal, operation role, exact path, and persistent identity; copying starts only after marker removal and directory sync complete. A corrupt, partially written, or mismatched marker cannot authorize automatic cleanup, so recovery fails closed. Explicit share or favorite mutations after delete completion block an older restore from overwriting newer intent. Delete revalidates the canonical Trash replica before removing the source stage, and restore revalidates the destination before removing the Trash source. A hard journal, participant, receipt, outbox, source, or destination failure returns `500`, preserves recovery evidence, and blocks later storage mutations even when the durable commit has already occurred. A persistence-only failure returns success with `workspace mutation persistence incomplete`; uncertain terminal journal cleanup retains or recreates a recoverable checkpoint and activates the recovery gate. Capacity cleanup after `completed` and reliable terminal journal cleanup uses `trash delete cleanup incomplete`. Recovery of `completed` retries only receipt completion, outbox acknowledgement, and journal cleanup and does not replay participant Apply. For one successful regular-file deletion without contention, the live Trash path performs twelve complete content-hash passes plus one copy read: two over the live file, four over the source stage, and six over the Trash destination. The permanent-delete path performs five complete content-hash passes: two over the live file and three over the source stage. Contention, failure handling, or rollback can add reads. A replaced post-rename stage is reported as `StagePath` only when a synchronized recovery copy passes identity and content revalidation; otherwise `InspectionPaths` identifies the locations for manual review. The storage lock and mutation epoch cover only in-platform operations; external filesystem writers and concurrent mounts remain outside that serialization boundary. Permanent deletion of an item already in Trash uses `prepared` and `committed` records under `.deleting`. Items with restore data receive a read-only participant preflight before physical mutation, and cleanup removes ownership by the exact original delete operation ID; a persistence failure retains `committed` evidence and activates the recovery gate. Startup initializes the share and favorite participant stores and hooks first, then recovers permanent deletions under `.deleting` followed by live transfers under `.transactions`; both passes finish before workspace-stage cleanup, background tasks, and network listeners. Recovery fails closed and prevents writable startup when journal, outbox, participant, content, or metadata evidence cannot be reconciled. The retention sweep removes both expired versions and expired Trash items, and capacity limits may remove Trash items before their persisted expiry.
- Emptying Trash now submits the exact confirmed ID set through `POST /api/v1/trash/empty`; the former `DELETE /api/v1/trash` endpoint has been removed. The confirmation dialog freezes items, IDs, count, and size when opened, and submits selections above 1000 items from that frozen set in batches. Deleted, remaining, and skipped outcomes from every valid response are retained. If a later batch fails or the server returns a partial result, the page refreshes Trash and keeps only items that remain in both the original frozen set and the current list as the retry scope. If that refresh fails while a request outcome is unknown, the page blocks another deletion until reconciliation succeeds; items already confirmed as deleted or skipped are still removed precisely from the local list. Items added after the dialog opened never enter the reconciliation or retry scope. Under one storage write lock, the server loads current items, preflights access rules for every selected item that still exists, and then permanently deletes in request order. Unselected, newly added, or already missing items cannot be deleted accidentally. The response completely partitions requested IDs into deleted, remaining, and skipped arrays and reports partial results separately from post-commit physical-cleanup warnings.
- Local backup manifests now use v2 and separately bind regular files and the exact `data/` directory topology, including empty directories and POSIX `rwx` permission bits. Setuid, setgid, sticky, and other special permission bits are rejected; ACLs, extended attributes, and ownership are outside this evidence scope. The structural config directory uses fixed mode `0700`. Local restore, restore drill, and `restore-verify` treat the trusted manifest as authoritative for directories instead of deriving a baseline from the snapshot's current tree; the final tree check rejects extra or missing entries, permission drift, and case-folding collisions. Restore records bind the same restore to a raw manifest-byte digest as internal evidence and do not return it as a public `RestoreResult` field. Explicit restore also rechecks the identity of its adjacent staging directory; on Unix, the staging directory and its ancestors must not be replaceable by an untrusted account. V1 snapshots cannot be restored or automatically pruned and are excluded from v2 retention counts, while the current snapshot and snapshots still referenced by restore history remain pinned. A new successful local backup is required for a v1 manifest or missing trusted evidence. This change applies only to `local` jobs; `restic` and `rclone` keep their backend-specific semantics. Native Windows cannot preserve the required POSIX permission bits losslessly, so v2 local operations are rejected there.
- Authenticated downloads now synchronize a short-lived download session and perform a bounded one-byte error probe. An expired access cookie reuses the existing refresh and cross-tab coordination flow, and concurrent synchronization for the same authentication generation and cancellation signal is coalesced. Requests that allow the download cookie require simultaneously present access and download cookies to have identical raw values, preventing an invalid current access cookie from falling back to a valid download cookie left by another account or session. Public-share downloads issue URL-safe, signed, target- and state-bound short-lived tickets of at most 256 characters through a same-origin POST. Both flows then use browser-native navigation for streaming instead of buffering complete files or ZIP archives as frontend Blobs. Download handoff uses a randomly named, sandboxed, hidden same-origin navigation context without a `src` attribute. The actual URL exists only briefly on a target link with `referrerpolicy="no-referrer"` and no `noopener` or `noreferrer`, preserving named-target resolution. Attachment responses use `sandbox allow-downloads`, `frame-ancestors 'self'`, and `X-Frame-Options: SAMEORIGIN`, so only the application itself can embed them, while inline previews continue to prohibit embedding. Successful navigation contexts are removed at the end of the page lifecycle, non-download error responses are removed after loading, and a limit of 64 active or pending contexts bounds per-page resource use. At that limit, a new submission is rejected before network probing or public-ticket issuance and instructs the operator to refresh the page. Browser/API download routes are exempt from the general 30-minute request-context deadline and advance the connection write deadline by `server.write_timeout` before response headers, each write, and each flush. An active transfer may therefore exceed that duration, while one stalled write still times out. Public-ticket requests require a persistent 32-byte `client_nonce` generated with Web Crypto. Under separate HMAC domains, the server derives a stable 128-bit binder ID and 256-bit binder value from the share ID and nonce, while each ticket retains an independent random ID. The ticket contains only the binder ID and binder-value digest. The same share and nonce refresh one `Path=/`, `HttpOnly`, `SameSite=Strict` cookie, while different shares derive different cookies. Cookie names end in the binder ID's 32-character lowercase hexadecimal encoding; HTTPS also uses `Secure` and the `__Host-` prefix. Download GET requests validate both the ticket query parameter and exactly one matching binder. Issuance applies a request-local soft limit of 32 structurally valid binder cookies: a valid existing target binder remains refreshable at or above the limit, while a new, forged, or duplicate target binder returns `429 Too Many Requests` with `Retry-After: 1` and `DOWNLOAD_TICKET_RATE_LIMITED`. A process-wide non-blocking capacity-4 issuance gate is held across ZIP preflight and bounds stale-cookie-snapshot overshoot from concurrent requests using different new nonces. `max_access` counts successfully issued logical download sessions; directory browsing, password validation, and Range or resumed transfers with the same ticket do not add another count. Password-failure throttling is partitioned by share and client address, permits one bcrypt check per key and 8 across the process, and retains at most 128 states per share and 4096 globally without evicting active failures, locks, or in-flight work. Passwords longer than 72 bytes do not consume state. Regular-file downloads and ZIP archives read validated regular-file snapshots only and reject FIFOs, sockets, and device files. Authenticated and public ZIP paths also verify each snapshot's actual byte count so truncated content is not represented as a complete archive. Authenticated ZIP probes and complete writes separately share one process-wide concurrency gate with capacity 4; saturation returns `429`, `Retry-After: 1`, and `ARCHIVE_DOWNLOAD_RATE_LIMITED` before filesystem access, and one Web UI batch submits at most 20 items with at most one directory. Public ZIP issuance preflight and actual ZIP streaming separately share another process-wide concurrency gate with capacity 4; a full gate returns the corresponding `429` response.

The public go-live smoke and doctor reject `localhost`, IP addresses, and four-part all-numeric hostnames, manual port-review commands include both connection and total request timeouts, and blank custom backend target lists or ambiguous target paths are rejected so port-exposure checks cannot be skipped or produce unclear backend probe URLs. The public go-live smoke only prints redacted target shapes for invalid custom backend targets and bad HTTP redirects, avoiding query strings, fragments, userinfo, and control-character path content in failure logs. ## Post-Publish Verification

After the release tag is published, prefer the unified go-live entry point:

```bash
./scripts/release-go-live-check.sh \
  --version <tag> \
  --domain nas.example.com \
  --repository seanbao/mnemonas \
  --artifact-dir dist/release-check \
  --backup-api-url https://nas.example.com/api/v1 \
  --backup-job-id external-disk \
  --cookie-file cookies.txt
```

To make the unified go-live check retain temporary downloaded artifacts for failure investigation, omit `--artifact-dir` and pass `--keep-published-artifacts`; explicit `--artifact-dir` values are maintainer-selected and already retained, so they cannot be combined with this option.
When the backup restore drill cannot be run for the release, pass `--skip-backup-restore-drill` explicitly and record that the release does not have complete restore evidence.
To verify only the GitHub Release artifacts, run:

```bash
mkdir -p dist/release-check
./scripts/verify-published-release.sh \
  --version <tag> \
  --repository seanbao/mnemonas \
  --artifact-dir dist/release-check
```

Then complete at least one archive-install smoke test, one Docker release-image startup smoke test, public documentation link checks, and deployment-environment review covering `mnemonas-doctor --public-domain`, external-network `public-go-live-smoke.sh`, DNS, firewall, TLS, and cloud security groups.
Explicit `--artifact-dir` values may use dash-prefixed relative paths, and repository names are validated as GHCR-compatible lowercase `owner/repo` values before download.
To retain temporary downloaded artifacts while investigating a failure, omit `--artifact-dir` and pass `--keep-artifacts`; the script prints the retained directory.

## Known Limitations

- There is currently no release candidate and no usable release has been published. The validation evidence in this document applies only to the named development branch and checkout; it does not indicate production readiness. The current source tree must not hold real data or be used for production deployment.
- The mountable SMB/Samba runtime is still not enabled. The current build only keeps configuration, diagnostics, and runtime-state notices.
- `LOCK` / `UNLOCK` are virtual WebDAV compatibility behavior. Concurrent multi-client edits of the same file still require conflict control in the client or upper workflow.
- Real public deployments depend on the specific DNS, firewall, TLS, reverse-proxy, and cloud security-group configuration. Templates and doctor checks do not replace environment-level review.
- If a future version introduces irreversible data migration, rollback should follow the matching release note or backup-restore procedure.
