## 定位与核心优势

### 一句话定位

**一个简单可靠、好看好用的私有 NAS** — 数据在自己手里，体验不输云服务。

### 核心优势：为什么选择 MnemoNAS？

#### 1. 数据自主权 (Ownership)

- **数据在自己手里**：没有订阅费、没有容量限制、没有隐私担忧
- **不依赖云服务**：服务商倒闭、涨价、改政策都与你无关
- **数据可迁移**：硬盘拔下来换台电脑就能用，不绑定特定软件

#### 2. 消费级体验 (Aesthetics)

- **现状**：开源 NAS (OpenMediaVault/Cockpit) 普遍是"工程师审美"，充斥着复杂的参数表单
- **我们**：采用 React + HeroUI + Tailwind 打造**类 iCloud/Synology Photos** 的沉浸式 UI
  - **相册模式**：自带高性能缩略图瀑布流，不再是冷冰冰的文件列表
  - **暗黑模式**：原生支持，细节打磨

#### 3. 性能 (Performance)

- **现状**：WebDAV 协议常被诟病"小文件慢、列目录卡"
- **我们**：
  - **智能预读**：利用 Rust 数据面高性能，对目录结构进行预加载
  - **秒传**：基于 CAS 的去重机制，相同文件瞬间完成"复制/上传"
  - **大文件优化**：流式传输优化，拖动 4K 视频进度条无缓冲感

#### 4. 功能 (Features)

- **文件版本历史**：CAS 机制自动保留每个文件的历史版本，支持回退
- **误删/误改可恢复**：回收站 + 版本历史，UI 上直接恢复
- **同步冲突无忧**：多设备同步出问题时，所有版本都在

#### 5. 整合能力 (Integration)

- **标准协议**：虽然底层是 CAS，但对外暴露标准的 WebDAV，完美兼容 Infuse, nPlayer, Windows Explorer, macOS Finder
- **API First**：所有功能都有清晰的 REST API，方便开发者编写脚本对接
- **Rclone 友好**：提供标准配置导出，轻松对接云存储做异地备份

#### 6. 使用 (Usage)

- **无感维护**：后台自动 Scrub（巡检）与 GC（垃圾回收），用户无需理解"存储池/卷/数据集"等概念
- **一键诊断**：出问题直接导出"诊断包"，包含脱敏日志与元数据统计，方便排查

## 技术选型

### 后端：Go 控制面 + Rust 数据面

| 组件 | 选型 | 理由 |
|------|------|------|
| **控制面语言** | Go 1.22+ | 生态成熟、部署简单、原生并发 |
| **数据面语言** | Rust 1.75+ | 极致性能、内存安全、无 GC 抖动 |
| **HTTP Router** | chi | 轻量、标准库兼容、性能好 |
| **WebDAV** | golang.org/x/net/webdav | 官方维护，成熟稳定 |
| **gRPC** | tonic (Rust) / grpc-go | 流式传输、强类型、高性能 |
| **哈希算法** | BLAKE3 | 比 SHA256 快 10 倍以上 |
| **CDC 分块** | FastCDC | 性能优于 Rabin，行业标准 |
| **日志** | zerolog (Go) / tracing (Rust) | 结构化、高性能 |

### 前端技术选型

| 组件 | 选型 | 理由 |
|------|------|------|
| **框架** | React 19 + TypeScript | 最新并发特性，生态成熟，AI 辅助友好 |
| **构建** | Vite | 极快的冷启动和 HMR |
| **组件库** | HeroUI | 开箱即有 Apple/Vercel 级别美观 |
| **样式** | Tailwind CSS 4 | 零运行时，设计系统一致 |
| **状态管理** | Zustand + TanStack Query | 极简 API，自动缓存 |
| **虚拟滚动** | @tanstack/react-virtual | 万级文件列表性能标杆 |
| **动画** | Framer Motion | 业界最佳动画库 |

### Go ↔ Rust 通信策略

| 方式 | 性能 | 开发成本 | 适用场景 |
|------|------|---------|---------|
| **gRPC** | 中 | 低 | ✅ 当前采用，大多数场景够用 |
| FFI/CGO | 高 | 高 | 热路径优化（未来） |
| Unix Socket | 中高 | 中 | 需要 zero-copy（未来） |

> **核心洞察**：NAS 场景的性能瓶颈通常在**磁盘 I/O**，而不是进程间通信。gRPC 的序列化开销在 SSD 场景下完全可接受。

## 架构决策

### 存储形态：Loose Object 模式

- **MVP 阶段**：1 个 Chunk = 1 个文件（松散对象）
- **理由**：利用文件系统原子性，避免自研 Packfile/GC 的早期复杂度
- **未来**：可按需引入 Packfile 优化小文件存储

### CAS 存储布局

```
objects/
├── ab/
│   └── cd/
│       └── abcd1234567890...  # 完整 BLAKE3 哈希作为文件名
```

- **2 层分片**：避免单目录文件过多（每层 256 个子目录）
- **原子写入**：写入 `.tmp` → `fsync` → `rename` → 目录 `fsync`
- **rclone 友好**：纯文件结构，可直接同步到云存储

### WebDAV 作为首发协议

| 协议 | 优点 | 缺点 | 决策 |
|------|------|------|------|
| **WebDAV** | 跨平台挂载容易、实现成本低 | 性能非极致 | ✅ 首发 |
| S3 | 开发者友好、生态工具多 | 挂载体验差 | 后续支持 |
| SMB/NFS | Windows/Linux 原生支持 | 实现成本极高 | 暂不支持 |

## 非目标与范围边界

### MVP 阶段明确不做

| 功能 | 原因 |
|------|------|
| SMB/NFS 协议 | 兼容性与实现成本过高 |
| RAID/复杂卷管理 | 先目录模式 + 清晰的数据目录约定 |
| 集群一致性 | 先单机可靠，再谈多机 |
| 去重作为强耦合 | 避免 GC/引用计数复杂度拖垮进度 |

### 数据安全承诺

**承诺**：
- 不因软件崩溃造成静默丢失
- 能发现静默损坏（端到端校验 + scrub）
- 误删可恢复（回收站 + 版本历史）

**不承诺**：
- 单盘物理损坏的数据冗余（需用户自行配置 RAID 或备份）
- 自动接管分区/格式化（用户提供已挂载目录）

### 竞争格局定位

- **vs 全功能 NAS (TrueNAS/OMV)**：不拼"功能列表"，拼"简单可靠 + 消费级体验"
- **vs 同步/备份工具 (Syncthing/restic)**：学习其恢复模型，但不急着覆盖所有场景
- **vs 对象存储 (MinIO/Ceph)**：MVP 不对标；后续从插件化开始
