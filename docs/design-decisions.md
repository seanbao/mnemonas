# 设计决策与技术选型

[English](design-decisions.en.md) | 简体中文

本文档记录 MnemoNAS 背后的主要设计选择，以及判断项目是否适合公开开源发布时使用的实践目标。

## 定位

MnemoNAS 是面向自托管部署的私有 NAS。它优先考虑数据所有权、可恢复性和经过打磨的日常 Web 使用体验，而不是追求大型设备式功能清单。

MnemoNAS 的目标：

- 数据所有权：不硬依赖云厂商。
- 可迁移存储：迁移完整存储根目录即可迁移当前文件、版本、回收站和元数据。
- 可用 Web UI：日常文件操作应像聚焦的产品，而不是运维后台。
- 可恢复性：回收站和版本历史内建在存储路径中。
- 标准访问：Web UI 面向人使用，WebDAV 面向常见客户端，REST API 面向自动化。
- 备份友好：完整存储根目录可用快照、restic、borg、rclone 或冷拷贝流程备份。

## 技术选型

### 后端

| 组件 | 选择 | 原因 |
| --- | --- | --- |
| 控制面 | Go 1.25.12+ | 部署简单、HTTP 生态成熟、并发支持强 |
| 数据面 | Rust 1.92+ | 适合存储密集逻辑的高性能和内存安全 |
| HTTP router | chi | 轻量、贴近标准库、行为可预测 |
| WebDAV | `golang.org/x/net/webdav` 加本地行为 | 以成熟基础实现为起点，补充项目特定兼容行为 |
| RPC | gRPC with grpc-go and tonic | Go 和 Rust 之间的强类型流式接口 |
| Hashing | BLAKE3 | 快速内容寻址和校验 |
| Chunking | dataplane file API 中的 FastCDC | Rust 已有实用内容定义分块；Go 版本存储当前使用整对象 CAS 快照 |
| Logging | zerolog and tracing | 低开销结构化日志 |

### 前端

| 组件 | 选择 | 原因 |
| --- | --- | --- |
| Framework | React 19 + TypeScript | 生态熟悉，类型支持强 |
| Build tool | Vite | 开发循环快，生产构建简单 |
| Components | HeroUI | 提供带主题支持的可访问组件基础 |
| Styling | Tailwind CSS 4 | 一致的 utility-first 样式，无运行时成本 |
| State | Zustand + TanStack Query | 轻量本地状态加服务端状态缓存 |
| Virtualization | TanStack Virtual | 支持大目录性能 |
| Animation | Framer Motion | 可控的交互打磨 |

界面目标是安静的文件管理工具：紧凑、易读，适合重复使用。它不应像宣传页，也不应像信息密集的管理后台。设置入口按使用目标组织，先显示状态和主要任务，再渐进披露低频网络参数；CDC、数据面连接等部署参数保留在配置文件和诊断信息中。运行时配置由对应任务页面承载：通知和磁盘健康归设备状态页，周期校验计划归备份与维护页，收藏开关归收藏页。各表单使用领域级局部保存，避免无关配置互相覆盖。

## 架构决策

### 原生当前文件加 CAS 版本

MnemoNAS 将当前文件保存为 `files/` 下的普通文件，并把历史版本保存到内容寻址对象存储中。

这避免把当前数据锁定在专有的纯对象布局中，同时仍让版本具备整对象去重和可校验能力。

### Packfile 之前先使用 Loose Objects

当前 CAS 布局在分片目录树下为每个对象保存一个文件：

```text
objects/
├── ab/
│   └── cd/
│       └── abcd1234567890...
```

原因：

- 用临时文件、`fsync` 和 rename 实现简单的崩溃一致写入。
- 可用常见文件系统工具直接备份和检查。
- 降低早期发布的实现风险。

如果对象数量或小对象开销成为实际问题，后续可以引入 packfiles。

### WebDAV 优先

| 协议 | 决策 | 理由 |
| --- | --- | --- |
| WebDAV | 首个支持的挂载协议 | 跨平台，且足够适合常见客户端 |
| S3 | 未来扩展 | 对工具有用，但不像挂载文件系统那样自然 |
| SMB/Samba | 仅保留配置预览 | 网关契约已经存在，但当前构建不启动 SMB 运行时 |
| NFS | 不在当前范围内 | 对小项目而言，权限和部署边界成本较高 |

## 发布质量目标

| 范围 | 目标 |
| --- | --- |
| 安装 | 新用户可在 10-15 分钟内访问 Web UI 和 WebDAV |
| 挂载 | 常见 macOS、Windows、Linux、移动端和 rclone 客户端有文档路径 |
| 大文件 | 覆盖流式下载、Range 请求和大文件上传路径 |
| 数据可信度 | Scrub 能报告缺失或损坏对象 |
| 恢复 | 可从 Web UI/API 使用回收站和版本恢复 |
| 诊断 | 可导出诊断包用于排查问题 |
| 崩溃一致性 | Kill/restart 测试不会留下半写入当前文件 |
| 认证 | 已实现 Web UI 用户、角色、密码和登录历史 |
| 分享 | 已实现密码、过期时间和公开访问流程 |
| 活动记录 | 重要用户操作可追踪 |

## 范围边界

当前非目标：

- SMB/NFS 协议栈。SMB 网关配置、凭据和授权目前仅作为预览性质的前置能力存在，当前构建不可挂载。
- RAID、文件系统创建、磁盘格式化或卷管理。
- 多节点一致性。
- 把去重作为核心用户可见功能。

数据安全承诺：

- 应用崩溃不应静默损坏当前文件。
- Scrub 应能报告缺失或损坏的 CAS 对象。
- 常见误删和误改应能通过回收站或版本恢复。

不承诺：

- 单盘物理故障下的数据冗余。
- 自动磁盘分区或格式化。
- 替代 3-2-1 备份策略。

## 竞争定位

MnemoNAS 不试图在功能数量上超过 TrueNAS 或 OpenMediaVault 这类完整 NAS 操作系统。它的目标是一个更小、更聚焦的自托管存储产品，提供短部署路径、可读的日常界面和明确的数据恢复工作流。

与 restic 或 borg 等备份工具相比，MnemoNAS 借鉴它们的恢复模型，但不试图覆盖所有工作流。与 MinIO 或 Ceph 等对象存储相比，它首先是私有 NAS。
