## 定位与范围

### 一句话定位

**自托管私有云存储** — 提供 Web UI、WebDAV、版本历史、回收站、Scrub 和诊断包，面向日常文件管理。

### 项目定位

#### 1. 数据自主权 (Ownership)

- **数据在自己手里**：不绑定云服务，容量由本机磁盘和备份策略决定
- **不依赖云服务**：服务商倒闭、涨价、改政策都与你无关
- **数据可迁移**：完整搬迁存储根目录即可换机运行，当前文件也保存在标准 `files/` 目录中

#### 2. 易用界面 (Usability)

- **问题背景**：很多自托管工具默认面向运维人员，常见问题是参数密集、层级深、移动端操作费力
- **界面目标**：采用 React + HeroUI + Tailwind 构建克制、清晰、可长期使用的私有云 UI
  - **清爽但不简陋**：保留品牌色、细腻边框、状态色和微交互，避免纯灰白后台感
  - **不做廉价炫技**：避免大面积玻璃、漂浮光球、强发光和过度紫蓝渐变，减少视觉疲劳
  - **面向真实操作**：文件、相册、版本、设置等页面优先保证可扫描、可点击、可恢复
  - **移动端可用**：常用入口在手机 Web 上可直接到达，不只是把桌面侧边栏压缩进小屏幕
  - **暗黑模式**：原生支持，细节打磨

#### 3. 性能 (Performance)

- **现状**：WebDAV 协议常被诟病"小文件慢、列目录卡"
- **实现方式**：
  - **智能预读**：利用 Rust 数据面高性能，对目录结构进行预加载
  - **去重复用**：基于 CAS 的内容寻址机制，相同内容可复用已有对象
  - **大文件友好**：读写路径使用流式传输，减少一次性内存占用，并保留后续性能基准空间

#### 4. 功能 (Features)

- **文件版本历史**：CAS 机制按策略保留适合版本化的文件历史，支持回退
- **误删/误改可恢复**：回收站 + 版本历史，UI 上直接恢复
- **同步冲突可追溯**：多设备同步出问题时，可从版本历史和回收站中排查并恢复

#### 5. 整合能力 (Integration)

- **标准协议**：虽然底层是 CAS，但对外实现 WebDAV RFC 4918 核心读写方法，并持续补充客户端兼容性矩阵
- **API First**：所有功能都有清晰的 REST API，方便开发者编写脚本对接
- **备份友好**：完整存储根目录可用 restic、borg、rclone 或文件系统快照做异地备份

#### 6. 使用 (Usage)

- **低心智维护**：Web UI 提供 scrub、GC 和健康诊断入口；版本保留会按配置自动清理
- **诊断包**：支持导出脱敏日志与元数据统计，方便排查问题

## 技术选型

### 后端：Go 控制面 + Rust 数据面

| 组件 | 选型 | 理由 |
|------|------|------|
| **控制面语言** | Go 1.25.9+ | 生态成熟、部署简单、原生并发 |
| **数据面语言** | Rust 1.92+ | 极致性能、内存安全、无 GC 抖动 |
| **HTTP Router** | chi | 轻量、标准库兼容、性能好 |
| **WebDAV** | golang.org/x/net/webdav | 官方维护，成熟稳定 |
| **gRPC** | tonic (Rust) / grpc-go | 流式传输、强类型、高性能 |
| **哈希算法** | BLAKE3 | 比 SHA256 快 10 倍以上 |
| **CDC 分块** | Rust dataplane 文件 API 使用 FastCDC | 当前 Go 版本历史路径仍使用 whole-object CAS 快照；chunk 级版本引用追踪尚未接入 |
| **日志** | zerolog (Go) / tracing (Rust) | 结构化、高性能 |

### 前端技术选型

| 组件 | 选型 | 理由 |
|------|------|------|
| **框架** | React 19 + TypeScript | 当前主流 React 技术栈，生态成熟，AI 辅助友好 |
| **构建** | Vite | 极快的冷启动和 HMR |
| **组件库** | HeroUI | 现代组件基础，便于统一交互、暗黑模式和可访问性 |
| **样式** | Tailwind CSS 4 | 零运行时，设计系统一致 |
| **状态管理** | Zustand + TanStack Query | 极简 API，自动缓存 |
| **虚拟滚动** | @tanstack/react-virtual | 万级文件列表性能标杆 |
| **动画** | Framer Motion | 业界最佳动画库 |

前端审美目标不是“营销站炫酷感”，也不是“工程师后台”。MnemoNAS 是高频使用的私有存储工具，界面应当像一个成熟的桌面/移动 Web 应用：安静、可靠、品牌清晰、长时间使用不累。

### Go ↔ Rust 通信策略

| 方式 | 性能 | 开发成本 | 适用场景 |
|------|------|---------|---------|
| **gRPC** | 中 | 低 | ✅ 当前采用，大多数场景够用 |
| FFI/CGO | 高 | 高 | 热路径优化（未来） |
| Unix Socket | 中高 | 中 | 需要 zero-copy（未来） |

> **核心洞察**：NAS 场景的性能瓶颈通常在**磁盘 I/O**，而不是进程间通信。gRPC 的序列化开销在 SSD 场景下完全可接受。

## 架构决策

### 存储形态：Loose Object 模式

- **当前实现**：1 个 CAS 对象 = 1 个 loose object 文件；版本历史保存 whole-object 快照
- **理由**：利用文件系统原子性，避免自研 Packfile/GC 的早期复杂度
- **未来**：可按需引入 Packfile 优化小文件存储

### CAS 存储布局

```
objects/
├── ab/
│   └── cd/
│       └── abcd1234567890...  # 完整 BLAKE3 哈希作为文件名
```

- **2 层分片**：避免单目录文件过多（每层 256 个子目录）
- **原子写入**：写入 `.tmp` → `fsync` → `rename` → 目录 `fsync`
- **备份友好**：对象与元数据都是普通文件，适合配合快照或冷备份同步到远端

### WebDAV 作为首发协议

| 协议 | 优点 | 缺点 | 决策 |
|------|------|------|------|
| **WebDAV** | 跨平台挂载容易、实现成本低 | 性能非极致 | ✅ 首发 |
| S3 | 开发者友好、生态工具多 | 挂载体验差 | 后续支持 |
| SMB/Samba | Windows/macOS 原生挂载 | 协议兼容性与安全边界复杂 | 仅保留网关预览配置，当前不启动 SMB 运行时 |
| NFS | Linux/Unix 原生挂载 | 权限模型和部署边界复杂 | 暂不支持 |

## 非目标与范围边界

### 当前范围明确不做

| 功能 | 原因 |
|------|------|
| SMB/NFS 协议栈 | 兼容性与实现成本过高；当前仅保留 SMB 网关配置、凭据和授权预览，不提供可挂载运行时 |
| RAID/复杂卷管理 | 先目录模式 + 清晰的数据目录约定 |
| 集群一致性 | 先单机可靠，再谈多机 |
| 去重作为强耦合 | 避免 GC/引用计数复杂度拖垮进度 |

### 数据安全承诺

**承诺**：
- 不因软件崩溃造成静默丢失
- 能发现静默损坏（端到端校验 + scrub）
- 误删可恢复（回收站 + 版本历史）

**不承诺**：
- 单盘物理损坏的数据冗余（需用户自行配置 RAID 或备份）
- 自动接管分区/格式化（用户提供已挂载目录）

### 竞争格局定位

- **vs 全功能 NAS (TrueNAS/OMV)**：不覆盖完整 NAS 功能列表，优先保证日常文件管理路径清晰
- **vs 同步/备份工具 (Syncthing/restic)**：学习其恢复模型，但不急着覆盖所有场景
- **vs 对象存储 (MinIO/Ceph)**：当前不对标；后续从插件化开始
