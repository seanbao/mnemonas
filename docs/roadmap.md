# MnemoNAS 路线图

[English](roadmap.en.md) | 简体中文

本文档记录 MnemoNAS 从“自托管文件云盘”走向“家庭/小团队 NAS”的产品路线。它用于指导开发优先级，不等同于发布承诺；具体范围会随数据安全、维护成本和用户反馈调整。

## 当前判断

截至 2026-06-03，MnemoNAS 已具备 Web 文件管理、WebDAV、版本历史、回收站、多用户、用户组、用户级容量配额、目录授权、分享、最近操作、健康检查、Scrub、GC、诊断包、Docker 和 systemd 部署等能力。它已经可以作为家庭/小团队的本地私有文件云盘试用，也可以用于非唯一副本的数据管理。

公网访问方向已有基础入口：反向代理脚本、`mnemonas-public-setup`、`mnemonas-doctor --public-domain`、Web 公网访问向导、安全自检 API、证书续期提示、证书失败排障引导、Web 设置页和 E2E 回归均已覆盖第一版流程。备份方向已有本地任务、restic/rclone 命令型远端目标、恢复演练/远端校验、恢复演练历史、成功率摘要与失败归因、轻量调度、自动备份时间窗、保留策略自动检测、本地、restic 与 rclone 恢复到安全目录、批量恢复预览与执行、恢复后只读校验与切换清单、恢复结果记录、恢复摘要发现项、匹配恢复校验关联、恢复演练过期提醒、维护页状态展示和 Webhook/Telegram/企业微信/SMTP 备份、恢复、恢复校验、恢复演练与保留策略通知。提醒方向已有保存后测试提醒入口，可通过 Web 设置页或 API 发送 `alert_test` 测试事件；目录授权和分享策略变更会产生 `settings_policy_changed` 事件，分享即将到期会产生 `share_expiring_soon` 聚合事件。磁盘健康方向已有基于 `smartctl` 的 SMART、温度、掉盘和序列号漂移检查，并接入设备状态页、诊断摘要、最近操作和 Webhook/Telegram/企业微信/SMTP 事件。配额方向已有管理员用户配额入口、`used_bytes` 动态统计、目录级配额配置、目录配额使用率展示、目录配额关注清单、目录配额保存前变更复核、非管理员 Web/API 写入 `home_dir` 的上传、复制、移动、回收站恢复、WebDAV 用户模式写入 `home_dir` 的 PUT/COPY/MOVE 用户配额硬限制，以及 Web/API 与 WebDAV 对目录配额的上传、复制、移动、恢复类写入拦截。权限方向已有用户组和目录授权，Web/API、WebDAV 用户模式、搜索、分享、收藏、回收站与活动过滤共用同一权限判定，并已在设置 API 和 Web 设置页提供有效权限检查。存储可见性方向已有文件系统容量、文件系统类型、挂载点、设备/数据集来源、脱敏挂载选项和原生数据校验提示。SMB 方向已有配置、凭据、网关授权与诊断预览，但当前版本仍不会启动可挂载的 SMB/Samba 运行时；该部分计划等 `smboxide` 项目成熟后再集成。但从完整 NAS 角度看，仍缺少 SMB 可挂载运行时、完整一键恢复流程、权限继承/分享规则整理、更多通知通道、更细的配额/权限管理视图和更完整的存储池管理。因此，在外部备份策略和公网安全检查流程稳定前，不建议把 MnemoNAS 作为唯一长期数据源，也不建议绕过 HTTPS 反向代理直接暴露到公网。

## 产品定位

MnemoNAS 的优先目标不是复刻 TrueNAS、Synology DSM 或 Unraid 的全部能力，而是成为一个可追踪、可迁移、以原生文件为核心的自托管文件系统入口。

优先级排序：

1. 数据安全和可恢复性优先于功能数量。
2. Web UI 与 WebDAV 体验优先于复杂存储池管理。
3. 可部署、可诊断、可备份优先于一键式全自动。
4. 对家庭/小团队真实工作流有明确价值的能力优先。

## 可用性分级

| 等级 | 目标 | 适用范围 | 必须满足 |
| --- | --- | --- | --- |
| L0 | 开发预览 | 本地调试、演示 | 基本文件读写、启动脚本、单元测试 |
| L1 | 私有文件云盘 | 个人或小团队非唯一副本；默认局域网或受控内网 | Web/WebDAV、认证、版本、回收站、部署文档、外部备份指南 |
| L1+ | 安全远程访问 | 通过域名或反向代理从公网访问 | HTTPS、反向代理预检、安全向导、cookie/CSRF 边界、登录防护、暴露端口检查 |
| L2 | 家庭 NAS 基线 | 局域网长期运行 | SMB、备份任务、SMART/磁盘提醒、通知、配额、恢复演练 |
| L3 | 小团队 NAS | 多用户协作与清晰分享记录 | 文件夹级权限、群组、分享规则整理、活动回顾、灾难恢复流程 |

当前项目整体接近 L1，并已具备 L1+ 公网安全入口的第一版基础。下一阶段优先硬化 L1+ 的公网安全检查流程，再推进 L2 的数据保护和局域网兼容能力。

### P3：高级能力

目标：增强长期使用体验，但不应阻塞 L2/L3 基线。

| 能力 | 方向 |
| --- | --- |
| 全文和媒体索引 | 文件名搜索扩展为文档全文、EXIF、视频元数据、重复文件扫描 |
| 同步客户端 | 提供桌面/移动同步或官方 rclone profile，解决离线同步体验 |
| 存储池管理 | 可视化 ZFS/Btrfs/mdraid 状态，必要时调用系统工具而不是重写存储栈 |
| 对象级加密 | 用户密钥、恢复密钥、密钥轮换、备份恢复流程 |
| 插件/自动化 | 基于扩展点增加任务 Runner、Webhook、媒体处理等能力 |

## 推荐实施顺序

1. **完善 HTTPS 和安全向导**：如果计划公网访问，继续降低证书、反向代理、cookie、CSRF、端口暴露、云安全组和默认账号的误配置风险。
2. **备份任务与恢复演练**：确保数据可恢复，再扩大真实使用范围。
3. **磁盘健康与通知**：继续补充 USB/RAID 桥接兼容性，并扩展通知通道。
4. **配额与权限模型**：继续补配额管理视图、趋势、ACL 和用户组，避免多用户长期运行时互相影响。
5. **SMB/Samba 集成**：等 `smboxide` 成熟后再接入可挂载运行时，补齐家庭 NAS 最常见的局域网访问方式。
6. **全文索引、同步客户端、插件**：在数据保护和权限模型稳定后推进。

## 质量门槛

每个 P0/P1 功能合入前至少满足：

- 单元测试覆盖核心边界条件。
- API/存储层集成测试覆盖成功、失败、回滚。
- Web E2E 覆盖桌面和移动端主流程。
- WebDAV 或 SMB 相关能力必须有客户端兼容性记录。
- 公网访问相关能力必须覆盖 HTTPS 证书状态、反向代理 header、Secure/SameSite cookie、CSRF、登录限速和内部端口未暴露检查。
- 涉及数据删除、恢复、备份、配额、权限的功能必须有故障注入或恢复测试。
- 文档必须说明使用边界、失败模式和恢复步骤。

## 非目标

近期不把以下事项作为主线：

- 替代完整 NAS 发行版的磁盘阵列管理。
- 在权限、版本历史、回收站和活动记录边界尚未完整验证前发布自研 SMB/NFS 协议栈。
- 无外部备份前承诺单点长期数据安全。
- 推荐绕过安全预检、doctor 检查和 HTTPS 反向代理的公网直连部署。
- 为公网匿名分享牺牲默认安全边界。

## 路线图维护规则

- 已完成的能力应移入 CHANGELOG 或对应功能文档，不长期堆在路线图中。
- 如果实现方式改变，应更新“验收标准”，而不是只改功能名称。
- 与数据安全相关的条目优先保留明确失败模式和回滚策略。
