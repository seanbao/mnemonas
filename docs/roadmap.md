# MnemoNAS 路线图

[English](roadmap.en.md) | 简体中文

本文档记录 MnemoNAS 从“自托管文件云盘”走向“家庭/小团队 NAS”的产品路线。它用于指导开发优先级，不等同于发布承诺；具体范围会随数据安全、维护成本和用户反馈调整。

## 当前判断

MnemoNAS 当前处于 L0 开发预览阶段，尚未发布任何可用版本。开发分支已实现 Web 文件管理、WebDAV、版本历史、回收站、多用户、用户组、用户级容量配额、目录授权、分享、最近操作、健康检查、Scrub、GC、诊断包，以及 Docker 和 systemd 路径验证等能力；这些功能和测试结果不构成可用版本或生产就绪声明。

当前能力可按以下方向概括：

- 路线图只记录产品判断、能力边界和下一步优先级。已完成硬化区域、验证证据和发布状态应以 CHANGELOG、发布说明草稿和对应功能文档为准，避免在路线图中重复维护完整完成清单。
- 公网访问已有第一版基础：反向代理脚本、`mnemonas-public-setup`、`mnemonas-doctor --public-domain`、Web 公网访问向导、安全自检 API、证书续期提示、证书失败排障引导、Web 设置页和 E2E 回归。公网诊断会检查后端端口的宽泛 UFW 放行规则，并支持对存储路径和 WebDAV 用户文件路径中的 `~` 做一致展开。
- 备份与恢复已有本地任务、restic/rclone 命令型远端目标、轻量调度、自动备份时间窗、保留策略自动检测、恢复演练或远端校验、恢复历史、恢复摘要、单任务恢复阶段进度、批量恢复阶段进度、恢复后只读校验、切换/回滚清单、可复制恢复切换记录、包含任务上下文、备份目标和保留策略状态的可复制批量恢复记录、批量恢复跨目录切换候选、批量恢复冲突处置记录、批量恢复待处理原因、首页风险摘要、维护页状态展示，以及 Webhook/Telegram/企业微信/钉钉/SMTP 备份、恢复、恢复校验、恢复演练与保留策略通知。
- 提醒和磁盘健康已有保存后 `alert_test` 测试入口、目录授权和分享策略变更事件、分享即将到期聚合事件，以及基于 `smartctl` 的 SMART、温度、掉盘和序列号漂移检查；这些状态已接入设备状态页、诊断摘要、最近操作和通知。
- 配额和权限已有用户级配额、目录级配额、实际用量统计、账号关注摘要、筛选、排序、复核摘要复制、当前清单导出、配额与权限联合复核视图、目录配额保存前复核，以及 Web/API/WebDAV 写入和恢复路径的配额限制。用户组和目录授权已在 Web/API、WebDAV 用户模式、搜索、分享、收藏、回收站、活动过滤和“用户管理 > 目录与访问”的有效权限检查中复用同一权限判定；分享路径策略也可按创建/维护者用户、组或角色限制。
- 存储可见性已有文件系统容量、文件系统类型、挂载点、设备/数据集来源、脱敏挂载选项、原生数据校验提示和管理员存储健康摘要。SMB 方向已有配置、凭据、网关授权与诊断预览，但当前开发构建仍不会启动可挂载的 SMB/Samba 运行时；该部分计划等 `smboxide` 项目成熟后再集成。

从完整 NAS 角度看，仍缺少 SMB 可挂载运行时、更多真实介质和远端恢复处置样例、更多真实权限和分享规则复核样例，以及更完整的存储池管理。当前源码不应承载真实数据、作为长期数据源或直接暴露到公网。

首页首次设置检查已改为服务端就绪状态：必需项直接验证管理员访问、初始密码更换、初始密码文件、安全阻断项、独立备份任务和当前有效的成功备份；建议项验证备用管理员、自动计划、恢复证据和安全警告。只有备份类必需项可以限期延期，完成与延期均持久化，并在写入前重新计算证据。该状态不替代 `mnemonas-doctor` 和云防火墙复核。

## 产品定位

MnemoNAS 的优先目标不是复刻 TrueNAS、Synology DSM 或 Unraid 的全部能力，而是成为一个可追踪、可迁移、以原生文件为核心的自托管文件系统入口。

优先级排序：

1. 数据安全和可恢复性优先于功能数量。
2. Web UI 与 WebDAV 体验优先于复杂存储池管理。
3. 可部署、可诊断、可备份优先于全自动化流程。
4. 对家庭/小团队真实工作流有明确价值的能力优先。

## 可用性分级

| 等级 | 目标 | 适用范围 | 必须满足 |
| --- | --- | --- | --- |
| L0 | 开发预览 | 本地调试、演示 | 基本文件读写、启动脚本、单元测试 |
| L1 | 私有文件云盘 | 个人或小团队非唯一副本；默认局域网或受控内网 | Web/WebDAV、认证、版本、回收站、部署文档、外部备份指南 |
| L1+ | 安全远程访问 | 通过域名或反向代理从公网访问 | HTTPS、反向代理预检、安全向导、cookie/CSRF 边界、登录防护、暴露端口检查 |
| L2 | 家庭 NAS 基线 | 局域网长期运行 | SMB、备份任务、SMART/磁盘提醒、通知、配额、恢复演练 |
| L3 | 小团队 NAS | 多用户协作与清晰分享记录 | 文件夹级权限、群组、分享规则整理、活动回顾、灾难恢复流程 |

当前项目处于 L0 开发预览阶段。开发分支已实现部分 L1 和 L1+ 目标能力，但尚未形成或发布可用版本。下一阶段先完成首个可用版本所需的数据安全、恢复和部署验证，再评估 L1 验收。

### P3：高级能力

目标：增强长期使用体验，但不应阻塞 L2/L3 基线。

## 推荐实施顺序

1. **完善 HTTPS 和安全向导**：如果计划公网访问，继续降低证书、反向代理、cookie、CSRF、端口暴露、云安全组和默认账号的误配置风险。
2. **备份任务与恢复演练**：确保数据可恢复，再扩大真实使用范围。
3. **磁盘健康与通知**：继续补充 USB/RAID 桥接兼容性，并扩展通知通道。
4. **配额与权限模型**：继续补配额管理视图、趋势、ACL 和用户组，避免多用户长期运行时互相影响。
5. **SMB/Samba 集成**：等 `smboxide` 成熟后再接入可挂载运行时，补齐家庭 NAS 最常见的局域网访问方式。
6. **全文索引、同步客户端、插件**：在数据保护和权限模型稳定后推进。

## 质量门槛

每个 P0/P1 功能合入前至少满足：

- 单元测试覆盖核心边界条件。
- API/存储层集成测试覆盖成功、失败、回滚。
- Web E2E 覆盖桌面和移动端主流程。
- WebDAV 或 SMB 相关能力必须有客户端兼容性记录。
- 公网访问相关能力必须覆盖 HTTPS 证书状态、反向代理 header、Secure/SameSite cookie、CSRF、登录限速和内部端口未暴露检查。
- 涉及数据删除、恢复、备份、配额、权限的功能必须有故障注入或恢复测试。
- 文档必须说明使用边界、失败模式和恢复步骤。

## 非目标

近期不把以下事项作为主线：

- 替代完整 NAS 发行版的磁盘阵列管理。
- 在权限、版本历史、回收站和活动记录边界尚未完整验证前发布自研 SMB/NFS 协议栈。
- 无外部备份前承诺单点长期数据安全。
- 推荐绕过安全预检、doctor 检查和 HTTPS 反向代理的公网直连部署。
- 为公网匿名分享牺牲默认安全边界。

## 路线图维护规则

- 已完成的能力应移入 CHANGELOG 或对应功能文档，不长期堆在路线图中。
- 已完成硬化区域和验证证据应记录在 CHANGELOG、发布说明草稿或对应功能文档中，路线图只保留会影响优先级判断的状态摘要。
- 如果实现方式改变，应更新“验收标准”，而不是只改功能名称。
- 与数据安全相关的条目优先保留明确失败模式和回滚策略。
