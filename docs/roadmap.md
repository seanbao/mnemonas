# MnemoNAS 路线图

[English](roadmap.en.md) | 简体中文

本文档记录 MnemoNAS 从“自托管文件云盘”走向“家庭/小团队 NAS”的产品路线。它用于指导开发优先级，不等同于发布承诺；具体范围会随数据安全、维护成本和用户反馈调整。

## 当前判断

截至 2026-05-09，MnemoNAS 已具备 Web 文件管理、WebDAV、版本历史、回收站、多用户、分享、活动日志、健康检查、Scrub、GC、诊断包、Docker 和 systemd 部署等能力。它已经可以作为家庭/小团队的本地私有文件云盘试用，也可以用于非唯一副本的数据管理。

公网访问方向已有基础入口：反向代理脚本、`mnemonas-public-setup`、`mnemonas-doctor --public-domain`、Web 公网访问向导、安全自检 API、Web 设置页和 E2E 回归均已覆盖第一版流程。备份方向已有本地任务、restic/rclone 命令型远端目标、恢复演练/远端校验、轻量调度、自动备份时间窗、保留策略、本地与 rclone 恢复到安全目录、维护页状态展示和 Webhook 失败通知。但从完整 NAS 角度看，仍缺少 SMB、完整一键恢复流程、磁盘健康告警、配额、细粒度权限、多通道通知和更完整的存储池管理。因此，在外部备份策略和公网安全检查闭环稳定前，不建议把 MnemoNAS 作为唯一长期数据源，也不建议绕过 HTTPS 反向代理直接暴露到公网。

## 产品定位

MnemoNAS 的优先目标不是复刻 TrueNAS、Synology DSM 或 Unraid 的全部能力，而是成为一个可审计、可迁移、以原生文件为核心的自托管文件系统入口。

优先级排序：

1. 数据安全和可恢复性优先于功能数量。
2. Web UI 与 WebDAV 体验优先于复杂存储池管理。
3. 可部署、可诊断、可备份优先于一键式全自动。
4. 对家庭/小团队真实工作流有明确价值的能力优先。

## 可用性分级

| 等级 | 目标 | 适用范围 | 必须满足 |
| --- | --- | --- | --- |
| L0 | 开发预览 | 本地调试、演示 | 基本文件读写、启动脚本、单元测试 |
| L1 | 私有文件云盘 | 个人或小团队非唯一副本；默认局域网或受控内网 | Web/WebDAV、认证、版本、回收站、部署文档、外部备份指南 |
| L1+ | 安全远程访问 | 通过域名或反向代理从公网访问 | HTTPS、反向代理预检、安全向导、cookie/CSRF 边界、登录防护、暴露端口检查 |
| L2 | 家庭 NAS 基线 | 局域网长期运行 | SMB、备份任务、SMART/磁盘告警、通知、配额、恢复演练 |
| L3 | 小团队 NAS | 多用户协作与审计 | 文件夹级权限、群组、策略化分享、审计报表、灾难恢复流程 |

当前项目整体接近 L1，并已具备 L1+ 公网安全入口的第一版基础。下一阶段优先硬化 L1+ 的公网检查闭环，再推进 L2 的数据保护和局域网兼容能力。

## 优先级路线

### P0：可靠私有云盘基线

目标：让 MnemoNAS 可以被谨慎地用于真实个人数据，但前提是用户已有外部备份。

| 能力 | 当前状态 | 下一步 | 验收标准 |
| --- | --- | --- | --- |
| 备份与恢复 | 已有备份指南、本地备份/恢复演练 API、本地快照恢复到安全目录、rclone 远端恢复到安全目录、restic/rclone 命令型远端目标、Web 维护页、轻量调度、自动备份时间窗、本地快照保留策略和 Webhook 失败通知；暂缺 restic 一键恢复编排 | 增加远端保留策略引导和 restic 恢复工作流 | 可配置本地目录、外置盘或 rclone/restic 目标；每个任务显示最近一次备份、恢复校验和告警状态 |
| 部署可靠性 | Docker/systemd 路径已存在 | 固化升级、回滚和配置迁移流程 | release 包安装、升级、卸载、数据保留均有自动化测试 |
| HTTPS 与安全向导 | 已有反向代理脚本、Traefik/Cloudflare Tunnel 模板、Web 向导、安全自检 API、doctor 公网/证书/HTTP 跳转检查、云安全组复核清单和桌面/移动端 E2E 回归 | 补齐证书续期提示和失败引导 | 公网域名、局域网自签、反向代理 header、Secure/SameSite cookie、CSRF、下载会话、内部 dataplane 端口未暴露均有自动化测试和文档 |
| 数据完整性 | 有 Scrub/GC/诊断 | 增加周期计划和失败通知 | Scrub 失败能在 UI、活动日志和通知中闭环 |
| 安全默认值 | Web 主会话使用 HttpOnly cookie | 增加登录限速、会话吊销、管理员安全提醒和危险配置告警 | 弱口令、默认密码、跨站请求、异常登录、公开分享误配置都有明确拦截或告警 |
| WebDAV 兼容性 | 已有基础矩阵 | 扩展 Windows/macOS/rclone 回归 | 关键客户端读写、重命名、删除、恢复均有记录 |

### P1：家庭 NAS 基线

目标：补齐家庭局域网 NAS 的基础能力。

| 能力 | 缺口 | 建议实现 | 验收标准 |
| --- | --- | --- | --- |
| SMB/Samba | 目前主要是 Web 和 WebDAV | 提供官方 Samba 集成方案，优先 systemd/Docker 配置生成 | Windows/macOS 可直接挂载；权限和路径映射不绕过安全边界 |
| 磁盘健康 | 有存储统计，缺少 SMART | 增加 SMART 采集、温度、坏盘、掉盘和剩余寿命状态 | UI 显示磁盘健康；异常进入活动日志和通知 |
| 通知系统 | Webhook 可覆盖磁盘空间与备份失败/警告；通道类型仍不足 | 支持邮件、Telegram/企业微信等通道，并接入更多事件源 | 磁盘满、备份失败、Scrub 失败、登录异常能触发通知 |
| 用户配额 | 多用户隔离不等于容量控制 | 增加用户/目录配额模型 | 上传、WebDAV 写入、复制、恢复都遵守配额 |

### P2：小团队协作

目标：让多用户协作更可控，降低误删、误共享和越权风险。

| 能力 | 缺口 | 建议实现 | 验收标准 |
| --- | --- | --- | --- |
| 文件夹级权限 | 目前以角色和 home_dir 为主 | 增加目录 ACL、群组和继承规则 | Web、API、WebDAV、分享均遵守同一权限判定 |
| 分享治理 | 已有分享链接 | 增加策略化分享、默认过期、下载次数限制、审计报表 | 管理员可查看、搜索、禁用所有分享 |
| 审计报表 | 已有活动日志 | 增加按用户、目录、分享、异常动作的报表 | 可导出审计数据，支持时间范围和操作类型过滤 |
| 管理员恢复工具 | 已有版本和回收站 | 增加跨目录恢复、批量恢复预检 | 恢复前显示冲突、覆盖和权限影响 |

### P3：高级能力

目标：增强长期使用体验，但不应阻塞 L2/L3 基线。

| 能力 | 方向 |
| --- | --- |
| 全文和媒体索引 | 文件名搜索扩展为文档全文、EXIF、视频元数据、重复文件扫描 |
| 同步客户端 | 提供桌面/移动同步或官方 rclone profile，解决离线同步体验 |
| 存储池管理 | 可视化 ZFS/Btrfs/mdraid 状态，必要时调用系统工具而不是重写存储栈 |
| 对象级加密 | 用户密钥、恢复密钥、密钥轮换、备份恢复流程 |
| 插件/自动化 | 基于扩展点增加任务 Runner、Webhook、媒体处理等能力 |

## 推荐实施顺序

1. **完善 HTTPS 和安全向导**：如果计划公网访问，继续降低证书、反向代理、cookie、CSRF、端口暴露、云安全组和默认账号的误配置风险。
2. **备份任务与恢复演练**：确保数据可恢复，再扩大真实使用范围。
3. **磁盘健康与通知**：让故障在数据丢失前暴露。
4. **SMB/Samba 集成**：补齐家庭 NAS 最常见的局域网访问方式。
5. **配额与权限模型**：避免多用户长期运行时互相影响。
6. **全文索引、同步客户端、插件**：在数据保护和权限模型稳定后推进。

## 质量门槛

每个 P0/P1 功能合入前至少满足：

- 单元测试覆盖核心边界条件。
- API/存储层集成测试覆盖成功、失败、回滚。
- Web E2E 覆盖桌面和移动端主流程。
- WebDAV 或 SMB 相关能力必须有客户端兼容性记录。
- 公网访问相关能力必须覆盖 HTTPS 证书状态、反向代理 header、Secure/SameSite cookie、CSRF、登录限速和内部端口未暴露检查。
- 涉及数据删除、恢复、备份、配额、权限的功能必须有故障注入或恢复测试。
- 文档必须说明使用边界、失败模式和恢复步骤。

## 非目标

近期不把以下事项作为主线：

- 替代完整 NAS 发行版的磁盘阵列管理。
- 自研 SMB/NFS 协议栈。
- 无外部备份前承诺单点长期数据安全。
- 推荐绕过安全预检、doctor 检查和 HTTPS 反向代理的公网直连部署。
- 为公网匿名分享牺牲默认安全边界。

## 路线图维护规则

- 已完成的能力应移入 CHANGELOG 或对应功能文档，不长期堆在路线图中。
- 如果实现方式改变，应更新“验收标准”，而不是只改功能名称。
- 与数据安全相关的条目优先保留明确失败模式和回滚策略。
