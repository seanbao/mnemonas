## 存储架构概述

MnemoNAS 采用 **原生文件 + 版本历史** 混合架构：当前用户文件直接存储为原生文件（保持可读性和兼容性），历史版本使用 CAS（内容寻址存储）保存。元数据通过 SQLite 管理，确保高效查询和事务一致性。

```text
┌─────────────────────────────────────────────────────────────┐
│                   MnemoNAS 应用层                            │
│  ┌───────────────────────────────────────────────────────┐  │
│  │  WebDAV 接口  →  统一存储层  →  版本管理  →  SQLite    │  │
│  └───────────────────────────────────────────────────────┘  │
│                                                             │
│  功能：原生文件访问、版本历史、完整性校验、软删除             │
├─────────────────────────────────────────────────────────────┤
│                   存储层                                     │
│  ┌─────────────────────┐    ┌─────────────────────┐        │
│  │  原生文件 (files/)  │    │  版本对象 (objects/)│        │
│  │  - 直接可读         │    │  - CAS 存储         │        │
│  │  - 当前版本可读      │    │  - 自动去重          │        │
│  └─────────────────────┘    └─────────────────────┘        │
├─────────────────────────────────────────────────────────────┤
│                   文件系统层（可选任意）                      │
│  ext4 / XFS / Btrfs / ZFS / NTFS / APFS                    │
├─────────────────────────────────────────────────────────────┤
│                   物理存储层                                 │
│  单盘 / RAID / 云存储                                       │
└─────────────────────────────────────────────────────────────┘
```

### 核心设计理念

1. **原生文件优先**：当前用户文件直接存储为原生格式，离线检查和迁移更直观
2. **版本历史分离**：历史版本存储在 CAS 中，不影响当前文件的可读性
3. **SQLite 元数据**：事务保证、高效查询、单文件易备份
4. **备份友好**：完整存储根目录可用快照、冷备份、restic、borg 或 rclone 备份；在线备份内部数据时优先使用文件系统快照

## 存储架构详解

### 目录结构

```text
~/.mnemonas/                        # 存储根目录（默认位于用户主目录）
├── files/                          # 用户文件（原生存储）
│   ├── documents/
│   │   └── report.docx             # 直接可用的原生文件
│   └── photos/
│       └── vacation.jpg
│
└── .mnemonas/                      # 内部数据（对用户隐藏）
    ├── index.db                    # SQLite 元数据库
    ├── objects/                    # 版本对象（CAS 存储）
    │   └── ab/
    │       └── cd/
    │           └── abcd1234...     # 历史版本内容
    └── trash/                      # 回收站
        └── {trash-id}/
            └── content             # 已删除文件内容
```

### 原生文件存储 (files/)

用户文件直接以原生格式存储，保持完整的文件系统语义：

```text
files/
├── documents/
│   ├── report.docx                 # Word 文档，可直接打开
│   └── notes.md                    # Markdown 文件
├── photos/
│   └── 2024/
│       └── vacation.jpg            # 照片，可直接查看
└── videos/
    └── family.mp4                  # 视频，可直接播放
```

**优势**：

- 具备操作系统目录权限时，`files/` 下的当前版本可以用普通文件管理器直接读取
- 通过 Web UI、WebDAV 或 API 写入时，才会触发版本历史、回收站和索引更新
- 不建议在服务运行时绕过 MnemoNAS 直接修改 `files/`；直接写入不会自动生成历史版本，直接删除也不会进入回收站
- 完整恢复版本历史、回收站和索引时，需要同时保留 `.mnemonas/` 内部数据并由 MnemoNAS 读取

### 版本对象存储 (objects/)

历史版本使用 CAS 存储，通过 BLAKE3 哈希寻址：

```text
objects/
├── ab/                             # 一级分片 (hash[0:2])
│   └── cd/                         # 二级分片 (hash[2:4])
│       └── abcd1234567890...       # 完整哈希作为文件名
├── 12/
│   └── 34/
│       └── 1234abcdef...
```

**特点**：

- 相同内容只存一份（自动去重）
- 读取时自动校验哈希（端到端验证）
- 原子写入保证崩溃一致性

**压缩说明**：Rust 数据面 CAS 支持 zstd 压缩（默认启用，超过 1KB 才会尝试压缩，且仅在压缩后更小才落盘）。压缩后的对象以 `.zst` 形式存储，不影响 BLAKE3 校验与去重逻辑。

### SQLite 元数据 (index.db)

使用 SQLite 管理所有元数据，包含以下表：

| 表名 | 用途 |
| ---- | ---- |
| `files` | 文件索引（路径、大小、修改时间、哈希） |
| `versions` | 版本历史（文件路径、版本哈希、时间戳） |
| `versioning_overrides` | 用户自定义版本策略 |
| `trash` | 回收站元数据 |
| `file_locks` | 文件锁（WebDAV LOCK 支持） |

**SQLite 优势**：

- 事务保证（ACID）
- 高效查询（索引支持）
- 单文件易备份
- 跨平台兼容

### 回收站 (trash/)

删除的文件移入回收站，保留原始内容和元数据：

```text
trash/
├── a1b2c3d4e5f6/                   # 回收站 ID
│   └── content                     # 原始文件内容
└── f6e5d4c3b2a1/
    └── content
```

回收站元数据存储在 `index.db` 的 `trash` 表中，记录原始路径、删除时间、过期时间等。

## 版本管理

### 版本策略

MnemoNAS 支持智能版本管理，可按文件类型和大小自动决定是否保存版本：

| 文件类型 | 默认策略 | 原因 |
| -------- | -------- | ---- |
| 文档 (.txt, .md, .doc, .xls) | ✅ 保存版本 | 编辑频繁，版本价值高 |
| 配置 (.json, .yaml, .toml) | ✅ 保存版本 | 配置变更需要可追溯 |
| 图片 (.jpg, .png, .gif) | ❌ 不保存 | 通常不修改，占空间 |
| 视频 (.mp4, .mkv) | ❌ 不保存 | 文件大，版本意义小 |
| 大文件 (>100MB) | ❌ 不保存 | 存储成本高 |

用户可通过 API 覆盖单个文件的版本策略。

### 版本保留策略

```toml
[storage.retention]
max_versions = 50           # 每个文件最多保留 50 个版本
max_age = "2160h"           # 版本最长保留 90 天
```

超出限制的旧版本会被自动清理，释放存储空间。

### 版本恢复

通过 REST API 查询和恢复历史版本：

```bash
# 查看文件版本历史
curl -H "Authorization: Bearer <access-token>" \
    http://localhost:8080/api/v1/versions/documents/report.docx

# 恢复指定版本
curl -X POST \
    -H "Authorization: Bearer <access-token>" \
    "http://localhost:8080/api/v1/versions/abc123.../restore?path=/documents/report.docx"
```

## 与传统 NAS 的区别

| 特性 | MnemoNAS 当前实现 | 传统 NAS (TrueNAS/Synology) | 纯 CAS 方案 |
| ---- | ----------- | -------------------------- | ---------- |
| **当前文件存储** | 原生文件 | 原生文件 | CAS 对象 |
| **版本存储** | CAS 对象 | 快照 | CAS 对象 |
| **文件可读性** | 当前文件可直接读，版本历史需 MnemoNAS 元数据 | 当前文件可直接读 | 需要专用软件 |
| **版本去重** | ✅ BLAKE3 | ⚠️ 依赖文件系统 | ✅ 完全去重 |
| **元数据管理** | SQLite | 文件系统 | JSON/DB |
| **复杂度** | 中 | 低 | 高 |

### 方案对比

相比纯 CAS 方案，当前混合架构的主要取舍：

| 方面 | 纯 CAS 方案 | 当前混合架构 |
| ---- | ------------- | -------------- |
| 当前文件 | CAS 存储 | 原生文件 |
| 当前文件离线读取 | 需要专用软件 | 可直接读取 `files/` |
| 元数据 | JSON 文件 | SQLite |
| 查询效率 | O(n) 扫描 | O(log n) 索引 |
| 事务保证 | 无 | SQLite ACID |

### 类似设计的项目

| 项目 | 架构 | 相似度 |
| ---- | ---- | ------ |
| **Synology** | 原生文件 + 快照 | ⭐⭐⭐⭐ |
| **Time Machine** | 原生文件 + 硬链接 | ⭐⭐⭐⭐ |
| **Git** | CAS 对象存储 | ⭐⭐⭐ |
| **restic** | 纯 CAS + 加密 | ⭐⭐ |

## 文件系统兼容性

**MnemoNAS 不依赖特定文件系统**，可运行在任何 POSIX 兼容系统上。

| 文件系统 | 兼容性 | 推荐度 | 说明 |
| -------- | ------ | ------ | ---- |
| **ext4** | ✅ | ⭐⭐⭐ | Linux 默认，稳定可靠，单盘首选 |
| **XFS** | ✅ | ⭐⭐⭐ | 大文件/高并发性能好 |
| **Btrfs** | ✅ | ⭐⭐⭐⭐ | 支持快照、scrub，可做额外保护层 |
| **ZFS** | ✅ | ⭐⭐⭐⭐⭐ | 自带 scrub、mirror、压缩，最推荐 |
| **NTFS** | ✅ | ⭐⭐ | Windows 环境可用 |
| **APFS** | ✅ | ⭐⭐⭐ | macOS 环境可用 |
| **exFAT** | ⚠️ | ⭐ | 无原子操作保证，不推荐 |
| **NFS 挂载** | ✅ | ⭐⭐ | 网络存储可用，注意延迟 |

## 最佳实践：底层配置推荐

### 推荐配置矩阵

根据预算和需求选择合适的底层配置：

| 场景 | 推荐配置 | 数据安全 | 性能 | 成本 |
| ---- | -------- | -------- | ---- | ---- |
| **入门/预算有限** | 单盘 ext4 + 云备份 | ⭐⭐ | ⭐⭐⭐ | $ |
| **家庭推荐** | ZFS mirror (2盘) | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | $$ |
| **进阶玩家** | Btrfs RAID1 (2盘) | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ | $$ |
| **兼容性优先** | mdadm RAID1 + ext4 | ⭐⭐⭐⭐ | ⭐⭐⭐ | $$ |
| **大容量** | ZFS RAIDZ1 (3盘+) | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | $$$ |
| **企业级** | ZFS RAIDZ2 + SSD 缓存 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | $$$$ |

---

### 配置 1：ZFS Mirror（推荐）

#### 最省心的高安全方案

```bash
# 创建镜像池（2块盘）
sudo zpool create mnemonas mirror /dev/sda /dev/sdb

# 设置挂载点
sudo zfs set mountpoint=/srv/mnemonas mnemonas

# 启用压缩（推荐 lz4，透明压缩）
sudo zfs set compression=lz4 mnemonas

# 设置记录大小（匹配 CAS 平均块大小）
sudo zfs set recordsize=1M mnemonas

# 定期 scrub（添加到 crontab）
# 0 2 * * 0 /sbin/zpool scrub mnemonas
```

**配置 MnemoNAS**：

```toml
[storage]
root = "/srv/mnemonas"
```

**优点**：

- 坏一块盘数据不丢失
- ZFS scrub 定期校验底层数据
- 透明压缩节省 20-40% 空间
- 企业级可靠性

---

### 配置 2：Btrfs RAID1

#### Linux 原生方案

```bash
# 创建 Btrfs RAID1（2块盘）
sudo mkfs.btrfs -m raid1 -d raid1 /dev/sda /dev/sdb

# 挂载
sudo mkdir -p /srv/mnemonas
sudo mount /dev/sda /srv/mnemonas

# 添加到 fstab
echo "UUID=$(blkid -s UUID -o value /dev/sda) /srv/mnemonas btrfs defaults,compress=zstd 0 0" | sudo tee -a /etc/fstab

# 定期 scrub
# 0 2 * * 0 /sbin/btrfs scrub start /srv/mnemonas
```

**优点**：

- Linux 内核原生支持
- 支持快照（可做额外备份层）
- 可在线扩容

---

### 配置 3：mdadm RAID1 + ext4

#### 经典兼容方案

```bash
# 创建 RAID1
sudo mdadm --create /dev/md0 --level=1 --raid-devices=2 /dev/sda /dev/sdb

# 格式化
sudo mkfs.ext4 /dev/md0

# 挂载
sudo mkdir -p /srv/mnemonas
sudo mount /dev/md0 /srv/mnemonas

# 保存配置
sudo mdadm --detail --scan | sudo tee /etc/mdadm/mdadm.conf
sudo update-initramfs -u
```

**优点**：

- 兼容性最好，常见 Linux 环境都能读
- ext4 成熟稳定
- 恢复简单

---

### 配置 4：单盘 + 云备份（最小方案）

#### 预算有限时的选择

```bash
# 使用现有单盘，ext4 即可
sudo mkdir -p /srv/mnemonas

# 从文件系统快照或停服务后的 /srv/mnemonas 备份到云存储
# crontab 中应先创建快照或进入冷备份窗口，再运行 rclone/restic/borg
```

**重要**：单盘方案必须配合云备份或外置盘备份，否则硬盘故障将导致数据丢失。

## 性能与安全指标

### 专业 NAS 数据安全标准

MnemoNAS 达到的数据安全指标：

| 指标 | 专业 NAS 标准 | MnemoNAS 实现 |
| ---- | ------------- | -------------- |
| **静默损坏检测** | ✅ 定期校验 | ✅ BLAKE3 端到端校验 + Scrub |
| **崩溃一致性** | ✅ 事务性写入 | ✅ 原子写入 (tmp→fsync→rename) |
| **误删恢复** | ✅ 快照/回收站 | ✅ 回收站 + 版本历史 |
| **版本回退** | ✅ 快照 | ✅ 每文件独立版本历史 |
| **数据去重** | ⚠️ 部分支持 | ✅ CDC 块级去重 |
| **备份验证** | ✅ 校验和 | ✅ 所有数据自带校验 |

### 数据安全层次

```text
┌─────────────────────────────────────────────────────────┐
│ 第 1 层：应用层保护（MnemoNAS 提供）                      │
│  • 端到端 BLAKE3 校验                                   │
│  • 原子写入保证                                         │
│  • 版本历史 + 回收站                                    │
│  • Scrub 定期巡检                                      │
├─────────────────────────────────────────────────────────┤
│ 第 2 层：文件系统保护（ZFS/Btrfs 提供）                  │
│  • 底层 scrub 校验                                     │
│  • 写时复制 (CoW)                                      │
│  • 文件系统级快照（可选额外备份）                        │
├─────────────────────────────────────────────────────────┤
│ 第 3 层：硬件冗余（RAID/Mirror 提供）                    │
│  • 单盘故障不丢数据                                    │
│  • 热备盘自动重建                                      │
├─────────────────────────────────────────────────────────┤
│ 第 4 层：异地备份（3-2-1 原则）                         │
│  • 云存储备份                                          │
│  • 物理外置盘                                          │
└─────────────────────────────────────────────────────────┘
```

### 性能基准

| 操作 | 基准 | 影响因素 |
| ---- | ---- | -------- |
| **顺序写入** | 接近磁盘原始速度 | 磁盘 I/O |
| **顺序读取** | 接近磁盘原始速度 | 磁盘 I/O |
| **小文件写入** | ~1000 文件/秒 | fsync 频率 |
| **目录列举** | ~10000 文件/秒 | 元数据 I/O |
| **去重命中** | 微秒级 | 内存索引 |
| **Scrub 速度** | ~100 MB/秒 | 磁盘顺序读 |

### 调优建议

**ZFS 调优**：

```bash
# 设置 ARC 缓存大小（服务器内存充足时）
echo "options zfs zfs_arc_max=8589934592" | sudo tee /etc/modprobe.d/zfs.conf  # 8GB

# 启用 SSD 缓存（L2ARC）
sudo zpool add mnemonas cache /dev/nvme0n1
```

**MnemoNAS 调优**：

```toml
[dataplane.cdc]
# 大文件为主时，增大平均块大小
avg_chunk_size = 2097152  # 2MB

[storage.retention]
# 减少版本保留以节省空间
max_versions = 20
max_age = "2160h"  # 90 天
```

## 总结

| 问题 | 答案 |
| ---- | ---- |
| **需要特定文件系统吗？** | 不需要，ext4 即可运行 |
| **推荐什么配置？** | ZFS mirror（2盘）最省心 |
| **能达到专业 NAS 安全标准吗？** | ✅ 结合 RAID + 备份可达到 |
| **与传统 NAS 有什么区别？** | CAS 架构，更接近 Git/restic |
| **数据能迁移吗？** | ✅ 完整搬迁存储根目录即可换机运行；当前文件可直接读，版本历史需 MnemoNAS 元数据 |

**关键原则**：

1. **应用层**：MnemoNAS 负责智能（去重、版本、校验）
2. **文件系统层**：推荐 ZFS/Btrfs 提供额外保护
3. **硬件层**：至少 2 块盘做镜像
4. **备份层**：遵循 3-2-1 原则，异地备份必不可少
