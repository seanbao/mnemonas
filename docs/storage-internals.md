# 存储原理与运维建议

[English](storage-internals.en.md) | 简体中文

本文档说明 MnemoNAS 的存储架构、它与传统 NAS 系统的差异，以及推荐的底层文件系统配置。

## 概览

MnemoNAS 使用混合布局：

- 当前用户文件保存为普通文件。
- 历史版本保存为内容寻址对象。
- SQLite 保存版本、回收站、锁和索引等元数据。

```text
+---------------------------------------------------------+
|                  MnemoNAS application                   |
| WebDAV/API -> storage layer -> versions -> SQLite       |
+---------------------------------------------------------+
|                     Storage root                        |
| files/      当前用户文件                                |
| .mnemonas/  元数据、CAS 对象、回收站                    |
+---------------------------------------------------------+
|                Underlying filesystem                    |
| ext4 / XFS / Btrfs / ZFS / APFS / NTFS                  |
+---------------------------------------------------------+
|                    Physical media                       |
| single disk / mirror / RAID / remote backup             |
+---------------------------------------------------------+
```

设计目标：

- 当前文件无需专用软件即可读取。
- 版本历史与当前文件分离。
- 一致性敏感的元数据使用事务性存储。
- 让完整存储根目录的备份和迁移保持直接。

## 目录布局

默认存储根目录：

```text
~/.mnemonas/
├── files/
│   ├── documents/
│   │   └── report.docx
│   └── photos/
│       └── vacation.jpg
└── .mnemonas/
    ├── index.db
    ├── objects/
    │   └── ab/
    │       └── cd/
    │           └── abcd1234...
    ├── write-staging/
    └── trash/
        ├── .deleting/
        │   ├── purge-{operation-id}.prepared.json
        │   ├── purge-{operation-id}.committed.json
        │   └── purge-{operation-id}.item
        ├── .transactions/
        │   ├── transfer-{operation-id}.prepared.json
        │   ├── transfer-{operation-id}.copying.json
        │   ├── transfer-{operation-id}.ready.json
        │   ├── transfer-{operation-id}.committed.json
        │   ├── transfer-{operation-id}.completed.json
        │   └── transfer-{operation-id}.item
        └── {trash-id}/
            └── content
```

## 原生当前文件

当前文件是 `files/` 下的普通文件。

优点：

- 具备操作系统级目录访问权限的用户可以直接读取当前文件。
- 离线迁移和备份更容易推理。
- 当前版本不会被锁定在专有的对象布局中。

重要边界：

- 读取 `files/` 是安全的。
- 在绕过 MnemoNAS 的情况下写入或删除 `files/`，不会创建版本、回收站记录、活动日志或元数据更新。
- 完整恢复版本、回收站和索引时，需要同时保留 `.mnemonas/` 和 `files/`。

### 写入暂存与发布边界

API 上传和 WebDAV PUT 会先把请求内容写入 `.mnemonas/write-staging/`。该目录不属于用户文件命名空间，也不会出现在普通文件列表中。服务端最多同时接收四个暂存写入，并对新分配的 source staging 字节与新建内容所需空间共用一个进程级 `10 GiB` 预算；该预算不是每个并发槽各 `10 GiB`。覆盖写入通过命名空间交换捕获的旧目标不重复占用运行时预算，因此事务期间 `write-staging` 目录的实际字节数可能超过 `10 GiB`。并发槽或进程级字节预算占满时，REST API 返回 `429 Too Many Requests`，WebDAV 返回 `503 Service Unavailable`，两者均提供 `Retry-After: 1`。

字节预留随请求体写入逐步增长，并在每次增长时检查暂存文件系统的 `AvailableBytes`、当前未落盘预留量和 `storage.retention.min_free_space`。无法取得容量信息，或可用空间不足以同时满足新增预留与最小剩余空间时，REST API 和 WebDAV 均返回 `507 Insufficient Storage`，且不提供 `Retry-After`。启动时会保守统计 `write-staging` 中所有遗留项的可计量字节，不区分其是否曾属于运行时 source staging 预算；只有经过完整复核的启动清理才能把该占用重置为零。新内容发布为正式文件或回滚暂存对象被确认删除后，对应预留才会释放。清理失败会保留预留量，并启用写入恢复门禁。

启动过程会在 `files/` 根目录与 `.mnemonas/write-staging/` 之间执行真实的双向禁止覆盖重命名探针和双向原子交换探针。禁止覆盖探针先确认已有目标不会被替换，再完成往返移动；交换探针确认两个现有目录项可在同一重命名域内原子互换。Linux 使用 `renameat2(RENAME_EXCHANGE)`，macOS 使用 `renameatx_np(RENAME_SWAP | RENAME_NOFOLLOW_ANY)`；缺少安全交换原语的平台会拒绝该布局。`EXDEV`、`ENOTSUP`、`EOPNOTSUPP`、`ENOSYS`、能力探针中的 `EINVAL`，以及任一身份复核、清理或同步失败，都会作为明确的原子写入布局错误阻止启动。

探针状态保存在 `InternalRoot/atomic-write-rename-probe-journal/`。该目录必须是未经过符号链接的 `0700` 目录，启动进程会在整个恢复和探测期间持有非阻塞独占 `flock`；同一存储根的另一个 MnemoNAS 进程无法并发执行探针。日志目录打开并通过身份和模式复核、独占锁取得后，启动流程会在任何恢复、日志或对象变更前无条件同步 `InternalRoot` 目录；即使日志目录已经存在，也会执行该父目录持久性屏障。严格 JSON setup intent 会先持久化两个随机 nonce、SHA-256 摘要、精确的 staging/files/isolation slot 和三个根目录身份，然后才创建探针对象。source 和 peer 对象分别以禁止覆盖方式创建在固定的 `pending-source.object` 和 `pending-peer.object`；完整写入 nonce 并同步文件后，流程会通过保留句柄和根目录相对打开结果复核身份与内容，再以禁止覆盖重命名发布到最终 isolation slot 并同步日志目录。对象身份绑定记录只会在该发布流程完成后写入。所有禁止覆盖重命名和原子交换操作均在操作前后写入不可变 phase checkpoint。每条 JSON 记录也会先写入固定 pending 文件并同步，再以禁止覆盖重命名原子发布为最终文件，最后同步日志目录；恢复会验证并发布完整的 pending 记录，不把部分写入暴露为已提交 phase。

启动发现未完成日志时，只检查日志给出的五个精确路径，并重新验证根目录、持久对象身份、模式、大小、修改时间、nonce 和摘要。恢复仅接受最后一个持久 phase 对应的布局，或该 phase 唯一的下一布局；对象会移入日志记录的确定性隔离 slot，再通过打开句柄身份复核和日志化的原地删除完成清理。恢复不会按 `.mnemonas-write-rename-probe-` 前缀扫描或删除 `files/` 中的其他对象。未知字段、截断或损坏的记录、未知目录项、符号链接、FIFO、同名替换、根目录替换、歧义布局及清理竞态均会保留现有对象并阻止启动。

该日志可恢复进程在已同步 checkpoint 后遭遇 `SIGKILL` 或崩溃的状态，包括 source 或 peer 对象创建、部分写入、文件同步和 isolation 发布后的窗口，以及对象位于 `files/` 的窗口；恢复和清理可重复执行。恢复仅在 setup intent 精确约束下处理 pending 探针对象：内容是预期 nonce 严格前缀的部分普通文件会在当前路径、身份和内容再次复核后执行受检删除并同步日志目录；完整对象会在同样的复核后以禁止覆盖方式发布并同步日志目录。存在多个 pending 对象、pending 与目标或绑定并存、特殊文件、未知内容或复核期间的同名替换时，现有对象会被保留并阻止启动。若进程在 pending JSON 记录尚未完整写入时终止，最终记录不会出现。完整且语义有效的 pending JSON 记录会在下次启动时继续发布；空文件、截断 JSON 或语法不完整的 pending JSON 记录从未构成已提交状态，恢复会通过打开句柄身份复核删除该记录，再从最后一个已提交 phase 继续。语法完整但 schema、身份或状态不匹配的 pending JSON 记录，以及符号链接、FIFO 或删除复核期间发生的同名替换，会被视为未知外部对象并保留，同时阻止启动。初始 setup intent 的 pending 文件不完整时尚未创建探针对象；后续 phase 的 pending 文件不完整时，最后一个已提交 phase 仍足以约束唯一恢复布局。持久性仍依赖操作系统和存储设备对文件及目录同步的保证；绕过权限和锁直接修改日志或探针 slot 的进程不在协作锁边界内，但后续身份复核会失败关闭。

根级探针不代表 `files/` 下的嵌套独立挂载。每次流式写入会在读取请求体和创建暂存对象前读取当前挂载表，并在取得全局变更锁后、发布工作区对象前再次复核。目标位于嵌套挂载点内或挂载表无法验证时，REST API 与 WebDAV 均返回 `503 Service Unavailable`。首次检查失败时不会读取请求体或修改目标；第二次检查失败时会清理已接收的暂存内容，且不修改目标。特权进程在第二次检查后并发改变挂载布局仍超出应用锁边界。最终跨根操作返回不支持的错误时，接口会报告原子写入布局错误。

流式写入要求直接父目录已经存在且是普通目录；REST 上传或 WebDAV PUT 不会隐式创建中间目录。该检查在读取请求体前完成，取得全局存储变更锁后还会再次验证。新内容在锁外写入内部 source stage，同时计算 BLAKE3 摘要并同步文件和暂存目录。服务端在读取请求体前记录目标快照，发布前重新核对目标身份、类型、模式、大小和删除令牌；读取期间发生的同名替换或内容变化会作为冲突处理。

每次写入在 `InternalRoot/write-transaction-journal/` 下使用不可变的 `prepared`、`published` 和 `committed` 检查点。日志目录是未经过符号链接的 `0700` 目录，进程会持有独占 `flock`。每个检查点包含操作 ID、前一记录摘要、完整计划摘要、存储根和父目录身份、目标前后状态、source/old-target 暂存证据、SQLite 元数据前后状态，以及需要时的 CAS 对象所有权。严格 JSON 记录先以禁止覆盖方式创建 pending 文件，完成写入和文件同步后原子发布，再同步日志目录；日志目录同步失败时不会把只在当前命名空间中可见的记录当作 durable decision。`prepared` 只在 source stage、根绑定、目标快照和元数据计划全部复核后发布，并且先于任何用户可见的命名空间变更。

新建文件通过跨根禁止覆盖重命名发布；覆盖写入直接把 source stage 与正式目标原子交换，不再在 `prepared` 前改名为另一个 exchange stage。交换后正式路径指向完整新文件，原 source 路径指向完整旧文件。目录同步顺序按当前持久决策保护权威对象：尚未提交的覆盖写入先持久化承载旧对象的 staging 父目录；前滚恢复优先持久化正式目标；任一首要同步失败时不会继续执行次要同步。应用内读取等待事务完成；绕过 MnemoNAS 的本地读取不受应用锁控制，但原子交换保证已有正式路径只呈现完整旧内容或完整新内容。可见发布前的取消会触发回滚；发布后的协调、回滚和恢复使用脱离请求取消的有界上下文。覆盖写入保留原权限位并清除 set-id 等特殊执行位。

可见发布后，服务端先按计划创建或验证旧内容 CAS 对象，再发布 `published` 结果并以单个 SQLite 事务确保文件索引和版本记录达到 after-state。只有命名空间、CAS 和元数据均确认后才发布 `committed`。`committed` 是唯一的前滚决策；缺少该检查点的事务一律回滚。恢复会按持久身份、类型、模式、大小、修改时间和 BLAKE3 摘要分类目标与 stage，只接受计划规定的 before/after 布局。CAS 创建、删除、引用检查和 SQLite 提交或回滚均可重试；操作产生副作用后返回不确定结果时，恢复会重新读取实际状态，而不会凭错误返回值推断结果。未知替换、额外目录项、身份漂移、摘要不符或参与者状态冲突会保留所有证据并阻止后续变更。

启动在 SQLite 恢复之后、后台任务和网络监听器启动之前完成写入事务恢复。若 SQLite 因损坏而重建，同时仍存在未决写入日志，启动会失败关闭，避免用不可靠元数据自动选择结果。恢复终态会重新同步 staging 和目标命名空间，复核文件、CAS 与元数据状态，再受检删除 stage 和日志；只有这些屏障全部成功后才释放暂存预算。通用 staging 清理会先扫描事务日志，存在未决或损坏记录时不会删除任何 source stage。恢复步骤可重复执行，并会继续处理与失败操作相互独立的其他日志；报告保留每个阻断操作及检查路径。持久性依赖操作系统和存储设备兑现文件与目录同步语义。绕过目录权限、生命周期锁和应用变更锁的同 UID 或特权进程不属于协作边界；后续复核会失败关闭，但无法阻止该进程在系统调用前后直接改写叶节点。

## CAS 对象

历史内容保存在内容寻址存储中：

```text
objects/
├── ab/
│   └── cd/
│       └── abcd1234567890...
```

属性：

- 使用 BLAKE3 哈希寻址。
- 相同内容可以复用同一个对象。
- 读取时校验哈希完整性。
- 写入使用临时文件、`fsync` 和 rename，以保证崩溃一致性。
- 当对象载荷压缩后更小时，支持使用 zstd 压缩。

## SQLite 元数据

`index.db` 保存如下元数据：

| 表 | 用途 |
| --- | --- |
| `files` | 文件索引数据 |
| `versions` | 版本历史 |
| `versioning_overrides` | 单文件版本策略覆盖 |
| `trash` | 回收站元数据 |
| `trash_operations` | 持久化回收站转移发件箱 |
| `file_locks` | WebDAV lock 状态 |

SQLite 为 MnemoNAS 提供 ACID 事务、索引和可迁移的元数据文件。

## 回收站

启用回收站时，删除的文件会移入 `.mnemonas/trash/`，元数据保存在 SQLite 中。元数据记录原始路径、删除时间、持久化到期时间和内容位置。关闭回收站时，删除会直接永久生效。

回收站到期天数和容量上限在 `[storage.trash]` 下配置。`[storage.retention].gc_interval` 驱动同一个后台任务清理过期文件版本与到期回收站项目。设为 `0` 会停用这两类周期清理，但不会停用容量清理、永久删除或清空回收站操作。

每个回收站项目在创建时取得随机 ID，该 ID 在项目生命周期内保持不变。精确清空操作在同一存储写锁内载入一次当前回收站状态，并在任何删除发生前，对所有仍存在的已选项目及其后代恢复逻辑路径完成访问规则预检。任一预检失败都不会删除已选项目。预检通过后，项目按请求 ID 顺序删除；未选择或在操作开始后新增的项目保持不变，已不存在的已选 ID 记为跳过。执行阶段出现硬失败时，尚未处理且仍存在的已选项目保持不变。删除、保留和跳过三个结果集构成原始请求的完整、互不重叠分区，并分别保留请求顺序。

### 回收站项目的永久删除恢复日志

显式永久删除、精确清空、到期清理和容量清理在永久移除已位于回收站的项目时，共用 `.mnemonas/trash/.deleting/` 下的持久化决策日志。若回收站项目包含分享或收藏恢复载荷，服务端会在扫描内容或发布日志前，以只读方式验证规范载荷、原始路径、参与者存储及恢复证据；验证失败不会改变内容、SQLite 元数据或日志。预检通过后，服务端写入并同步 `purge-{operation-id}.prepared.json`；记录包含完整回收站元数据、按路径排序的内容清单和待处理的版本对象摘要。内容随后使用禁止覆盖的重命名移入同一目录的 `purge-{operation-id}.item`。SQLite 回收站元数据移除后，服务端再写入并同步内容一致的 `purge-{operation-id}.committed.json`，然后才递归清理暂存内容、版本元数据和不再被引用的版本对象。物理与版本清理完成后，服务端按恢复载荷中的原删除操作 ID 精确清除已完成的分享与收藏所有权；该持久化屏障确认后才移除 purge 日志。

启动恢复在保留清理任务、工作区暂存清理和网络监听器启动前执行。任何恢复变更发生前，流程会对全部带恢复载荷的 `committed` 操作执行全局只读参与者预检；任一失败都会阻止本轮全部 `prepared` 回滚和 `committed` 前滚。仅存在 `prepared` 记录时，恢复流程验证完整清单，将内容回滚到规范回收站路径，并在缺失时恢复 SQLite 元数据。恢复流程先完成本次扫描中的全部 `prepared` 回滚；任一回滚失败或上下文取消都会阻止全部 `committed` 前滚，以便在版本清理前恢复共享版本所有权证据。存在 `committed` 记录时，操作只能前滚：删除仍存在的回收站元数据和暂存内容，完成版本清理，通过参与者持久化屏障清除精确所有权，最后移除决策日志。首次参与者持久化警告会重试一次；持续警告或硬失败会保留 `committed` 日志并启用恢复门禁。已删除的清单条目视为前滚的已完成部分；额外条目、同名替换物或身份变化不会被清理。`committed` 决策不会回滚为可恢复项目。

日志文件名、操作 ID、决策对、清单、暂存内容或 SQLite 元数据之间无法安全互证时，恢复以 fail-closed 方式终止可写启动，并保留相关路径供人工检查。运行期回滚无法完成时，后续存储变更也会被阻止，直至恢复成功。符合新命名格式但没有决策日志的孤立 `.item` 会阻止启动恢复。无法识别的旧版残留、临时文件和其他未跟踪路径只会报告并保留；文件名模式本身不构成自动删除授权。

该日志仅覆盖已位于回收站的项目被永久删除的路径。实时“移入回收站”删除和“从回收站恢复”操作使用下述独立协议。

### 实时回收站转移恢复日志

实时“移入回收站”删除和“从回收站恢复”操作使用 `.mnemonas/trash/.transactions/` 下的持久化旁路检查点。操作会按执行进度依次发布 `prepared`、`copying`、`ready`、`committed` 和 `completed` 记录，并可能保留私有副本 `transfer-{operation-id}.item` 或源端同级暂存路径 `.mnemonas-trash-transfer-{operation-id}.stage`。在发布 `copying` 前，新建的操作专属容器以及恢复流程新建的父目录会包含权限为 `0600` 的规范化 `.mnemonas-trash-transfer-owner-{operation-id}` 标记；若派生名称与恢复目标的任一父路径组件相同，操作 ID 分配器会拒绝该候选并重新分配。该标记绑定 `prepared` 日志哈希、操作 ID、角色、精确相对路径和持久身份。`copying` 在载荷复制前持久化已同步的容器与父目录身份；标记移除及其所在目录同步均完成后，载荷复制才会开始。标记不会进入最终回收站内容或恢复目标。只有副本完成并通过完整清单验证后才能发布 `ready`。记录包含存储根身份、逻辑路径、源清单与副本清单，以及参与者载荷。日志正文使用 SHA-256 哈希且不包含决策字段，因此正文一致的检查点会绑定到同一操作证据。

SQLite `trash_operations` 发件箱行与回收站元数据及文件索引变更在同一事务内写入，并绑定操作 ID、回收站 ID、操作类型、参与者载荷和日志哈希。分享与收藏参与者在各自配置的存储中保存操作级回执和原删除操作所有权。相同操作的重放幂等；删除完成后的显式分享更新、删除或收藏变更会阻止旧恢复操作覆盖新意图。转移达到 `completed` 后只完成转移回执，删除所有权会保留到对应回收站项目成功恢复或永久清理；永久清理仅依据 purge 日志载荷中的精确删除操作 ID 移除所有权。

通过预检且没有匹配发件箱行的 `prepared`、`copying` 或 `ready` 链执行回滚。`prepared` 恢复会根据完整且匹配的 owner marker 清理新建的容器与父目录；若崩溃发生在私有容器创建后、marker 写入前，则只自动回收精确操作路径上权限为 `0700` 的空容器。缺少 marker 的恢复父目录会保留，日志仍可回滚；部分写入、损坏、非规范化或不匹配的 marker，以及包含未知条目的容器，均无法证明所有权，恢复会保留证据并要求人工核对。`copying` 恢复只会清理身份与清单均属于该操作的部分副本。带匹配发件箱行的 `ready` 链和所有通过预检的 `committed` 链只前滚。`completed` 链不再重放参与者 Apply，只验证文件系统终态，并重试回执清理、发件箱确认和日志清理。检查点链无效、发件箱行缺失或不匹配、参与者证据不可靠、存在未跟踪的归属暂存对象，或身份与清单发生漂移时，恢复以 fail-closed 方式停止并保留证据供人工检查。

启动时先初始化 API 使用的分享与收藏参与者存储及钩子，随后依次恢复 `.deleting` 下的永久删除和 `.transactions` 下的实时转移。两类恢复都在工作区暂存清理、后台任务和网络监听器启动前完成。恢复失败会阻止可写服务启动；运行期出现需要恢复的失败后，后续存储变更会被阻止，直至恢复成功。若终态日志已移除但父目录同步失败，服务端会尽力重新持久化 `completed` 日志；回滚终态出现同类不确定性时则重新持久化规范化 `prepared` 日志。两种情况都会启用恢复门禁。可见变更已提交时，当前请求仍可返回持久化警告，但在恢复确认终态并清除日志前，不会执行容量清理或接受新的存储变更。

分享或收藏持久化文件因损坏而被隔离时，服务端会在对应存储文件旁写入 `<存储文件>.recovery-required` 标记。该标记跨进程重启保留“参与者恢复证据不可靠”状态；缺失或重新生成的主存储文件不会自动清除该状态。只有在依据隔离副本、备份与待处理回收站日志完成核对后，运维人员才能移除该标记并重新执行启动恢复。

日志恢复使用由设备号、inode 和对象类型组成的持久身份，并分别复核权限、大小和文件内容摘要。当同一文件系统内的重命名保持这类身份时，Linux 和 macOS 可执行日志恢复和基于目录句柄的逐项删除；其他平台会在发布清理记录、移动回收站项目或修改其元数据前拒绝该操作。重新挂载、恢复快照或转移卷后，设备号可能漂移；即使 inode 和内容仍一致，恢复也会拒绝将其视为原对象，并要求人工核对。

安全恢复还要求每个存储根目录在任一时刻只有一个 `nasd` 写入者。MnemoNAS 存储锁是进程内锁，不提供跨进程单写者仲裁；同一存储根目录上同时运行多个 `nasd` 实例不受支持。

Linux 与 macOS 上，工作区元数据读取会根据设备号、inode、ctime、类型与权限位、大小和纳秒级修改时间生成不透明对象身份令牌。REST 删除确认要求调用方提交从当前列表取得的身份令牌。服务端在短暂持有存储读锁时采样完整删除策略与变更纪元（`mutation epoch`），随后释放锁，在锁外对请求中的全部目标树执行严格遍历和普通文件哈希。扫描完成后，服务端再次核对变更纪元；若其发生变化，则丢弃整批扫描结果并执行有限重试，必要时在存储读锁保护下完成保底扫描。根对象回调依次检查写权限、工作区根目录下的嵌套挂载边界、文件类型和观察身份；身份不匹配时不会读取内容或遍历目录。通过根对象检查后，遍历会对每个后续条目执行相同的权限、挂载边界和类型检查，再通过不跟随符号链接且只接受普通文件的读取路径计算内容哈希，并按名称排序的深度优先顺序将节点摘要流式折叠为 v3 目标令牌。目标树中的嵌套挂载点、符号链接、FIFO、Unix socket 和其他特殊文件会被立即拒绝，不会形成不完整的目标令牌，也不会在打开阶段阻塞。挂载边界来自主机挂载表，因此同一设备上的 bind mount 也会被识别；目标树包含挂载点或目标位于挂载点内都会被拒绝。v3 目标令牌采用分层 SHA-256 Merkle 表示。最终令牌绑定规范化根路径与完整路径层级。每个节点摘要绑定文件或目录的域分隔标记、完整权限模式（`mode`）、大小、纳秒级修改时间、对象身份和快照中的内容哈希字段；普通文件的该字段承载实际内容摘要。目录节点按子项名称排序，并依次组合子项名称与摘要。最终目标令牌仍为 64 字符小写十六进制字符串，调用方必须将其视为不透明值。目录与空目录都会参与计算。删除确认结果只保留 REST 响应所需的根元数据和目标令牌，不保留完整目标树清单。单次确认最多包含 1000 个互不嵌套的目标。平台无法提供所需对象身份时，列表返回 `null`，且服务端拒绝创建观察式删除意图。无竞争时，每个目标仍只扫描一次，每个普通文件仍只计算一次内容哈希。发生竞争时，被丢弃的尝试会增加相应的扫描与哈希次数。变更纪元不参与 v3 目标令牌计算，也不出现在 REST 请求或响应中；现有 REST 请求与响应结构保持不变。

执行删除时，完整删除策略比较、当前目标树的逐项写权限复核、目标令牌比较与变更操作在同一存储写锁内完成。策略比较先于目标树遍历，过期策略不会触发全树读取。在对象捕获开始前发现删除方式、保留天数、清理周期、容量上限或目标树变化时，不会提交工作区、索引、版本、分享、收藏、回收站或活动变更；已确认目标消失或父路径不再是目录也按目标变化处理。对象捕获已经开始后，失败路径可能执行禁止覆盖的回滚，并可能改变对象 ctime 或父目录时间戳。WebDAV 条件删除会先在同一写锁内复核完整目标树的写权限，再读取相关目标属性并求值条件；只有依赖 ETag 的条件才计算内容哈希。

权限、目标令牌或 WebDAV 条件通过后，删除流程不会再次从原逻辑路径解析待删除对象。服务端先通过不跟随符号链接的句柄打开根对象作为见证对象，将句柄身份与本次请求中已验证快照的根身份比较，并在快照未包含摘要时为普通文件见证捕获内容哈希；随后使用禁止覆盖的原子重命名，将当前叶节点捕获到源文件系统同一父目录下的随机暂存路径。暂存对象必须与见证对象指向同一对象。若改名后的暂存路径在复核前被未知同名对象覆盖，服务端仅在仍打开的普通文件见证能够生成独立副本、同步文件与父目录，并再次通过路径身份与内容哈希校验时，才把独立恢复副本写入残留路径；未知对象保持原位。目录见证或无法完成同步与验证的普通文件恢复不会把未知暂存路径或未确认副本标记为恢复证据；`StagePath` 保持为空，实际未知路径通过 `InspectionPaths` 和错误日志提供人工检查。完整暂存树会映射回原逻辑路径，并再次核对条目集合、类型、大小、修改时间、后代身份与文件内容。文件哈希前后也会复核打开句柄和暂存路径仍指向同一对象。重命名可能改变根对象的 ctime，因此仅在 `os.SameFile` 验证成功后沿用变更前的根身份；后代身份仍按暂存状态逐项比较。

普通文件副本发布后，复制流程会保持目标文件句柄打开，核对复制字节数，并完整哈希已发布目标，将结果与复制读取期间累积的摘要比较；句柄与目标路径在哈希前后也必须保持同一身份。完整复制证明覆盖路径、类型、大小、权限、对象身份和摘要；不读取文件内容的遍历会在持久业务状态提交前再次复核两端目录树及其挂载边界。

实时“移入回收站”删除只会在写入 `ready` 检查点后，才把私有副本发布到规范回收站路径。随后，SQLite 事务原子更新回收站元数据与索引并插入发件箱行。已提交参与者状态应用完成后，服务端会在基于句柄清理源暂存对象前立即复核规范回收站副本。硬失败会保留日志与发件箱证据供前滚恢复，并阻止后续存储变更，不会降级为清理警告。永久删除仍会把已验证源暂存对象移入同一父目录下权限不宽于 `0700` 的随机隔离目录，再通过服务端持有的目录句柄逐项移除。未知条目、同名替换、身份变化或新挂载都会停止永久清理并保留隔离内容。

在无竞争且成功的单个普通文件删除中，删除确认与提交调用的实时回收站路径执行 12 次完整内容哈希加 1 次复制读取：当前文件 2 次、源暂存对象 4 次、回收站目标 6 次。永久删除路径执行 5 次完整内容哈希：当前文件 2 次、源暂存对象 3 次。竞争、失败处理或回滚可能增加文件读取次数。仅在改名后暂存路径被替换的见证恢复分支中，`StagePath` 才只指向已验证的恢复副本；无法确认时该字段保持为空，`InspectionPaths` 列出未知暂存对象和未确认副本的实际位置。其他失败阶段的 `StagePath` 表示已保留的内部残留位置，仍需结合错误原因和文件系统证据人工核查。

其他内部迁移路径使用的通用跨根普通文件移动会核对源路径初始身份、打开句柄身份、实际复制字节数、复制后句柄状态和当前路径身份，并用 `.mnemonas-move-*.tmp` 与 `.mnemonas-remove-*` 隔离路径只清理仍与复制证明一致的对象。回收站恢复不使用该通用隔离流程，而是先创建并同步专用工作区转移容器、写入 `copying`，再把规范回收站源复制到该容器的载荷路径；完整副本验证通过后写入 `ready`，禁止覆盖地发布目标，再把回收站、索引、版本、策略覆盖和文件锁变更与发件箱行一并提交。移除规范回收站源之前，流程会立即复核完整目标清单；目标发生漂移时恢复会被阻止，并在回收站源仍存在时保留该源。

回收站删除只从已验证的暂存路径复制内容，并保留该源暂存对象直至持久业务状态提交。只有已提交参与者状态应用完成且规范回收站副本再次通过验证后，才会移除源暂存对象；同文件系统和 `EXDEV` 路径采用相同流程。提交前只对已验证暂存对象执行禁止覆盖的回滚；提交后恢复只允许前滚。永久删除也只清理暂存对象，不会再次从原逻辑路径解析并删除。删除期间原路径出现的新对象不会被复制、覆盖或移除。回滚无法验证时，新对象保持不变，REST 返回 `500 Internal Server Error`，且不记录删除活动。

永久删除模式在逻辑提交后发生物理清理失败时，REST 返回带 `delete cleanup incomplete` 的 `200 OK`，WebDAV 返回带同一警告的 `204 No Content`；逻辑路径保持已删除。实时回收站模式只有明确归类为持久化不完整的参与者或回执失败才会返回成功，并附带 `workspace mutation persistence incomplete`。转移达到 `completed` 后才执行容量清理；容量清理失败返回成功并附带 `trash delete cleanup incomplete`。日志、参与者、回执、发件箱、源端或目标端发生硬失败时返回 `500 Internal Server Error`；该失败可能发生在持久提交后，恢复证据会被保留，存储变更门禁也会启用。恢复 `completed` 操作时只重试完成步骤，不重放参与者 Apply。

上述原子边界与变更纪元只覆盖通过 MnemoNAS 存储锁执行的操作。同 UID 进程直接修改文件系统或特权进程并发挂载既不会推进该纪元，也不受存储锁约束。对象身份、挂载边界、暂存对象和恢复安全检查会拒绝已观察到的变化，但最后一次复核与文件系统调用之间发生的变化仍超出应用级事务边界。`.deleting` 日志覆盖已位于回收站的项目在永久删除期间发生的进程崩溃和断电；证据有效时，`.transactions`、`trash_operations` 和参与者回执覆盖实时“移入回收站”删除与“从回收站恢复”期间的同类故障。服务端不会移动或删除身份未知的替换物，也不会自动清理未经验证的 `.mnemonas-delete-*`、`.mnemonas-trash-transfer-*`、`.mnemonas-move-*`、`.mnemonas-remove-*` 或回收站内部残留。`InspectionPaths` 与服务端日志会列出已知位置；不属于可验证恢复链的残留需要依据这些路径和文件系统证据人工复核。

删除专用遍历不会改变普通目录遍历、搜索或文件计数行为。删除流程在意图快照、权限复核、回收站目标描述、跨根复制和受检清理时分别重新读取挂载表。读取失败或挂载路径非法时会拒绝继续操作并返回 `ErrNotRegular`。尚未捕获目标时，REST 和 WebDAV 将其映射为 `409 Conflict`。永久删除模式下，捕获后无法安全回滚会形成恢复残留，逻辑提交后的失败则形成清理警告。实时回收站模式在检查点发布后发生硬失败时会保留持久证据、返回 `500`，并要求先恢复再执行后续存储变更。仅在目标边界仍可验证时清理已复制副本；边界无法验证时会保留内部副本，以免跨越新挂载。回收站项目的永久暂存和递归清理采用相同边界规则。

每个回收站项目的到期时间在删除时确定，后续配置变化不会重新计算。容量不足时，较早项目仍可能在到期前被清理，因此到期时间不是最低保留保证。

## 版本策略

MnemoNAS 会自动为通常有历史价值的文件保存版本：

| 文件类型 | 默认行为 | 原因 |
| --- | --- | --- |
| 文本、Markdown、办公文档 | 保存版本 | 经常编辑 |
| 配置和源码文件 | 保存版本 | 变更应可追踪 |
| 图片 | 默认不保存版本 | 通常体积较大且追加式保存 |
| 视频 | 默认不保存版本 | 体积很大 |
| 超过默认大小限制的文件 | 默认不保存版本 | 存储成本高 |

整文件版本对象受当前数据面一元写入契约限制，硬上限为 `104857600` 字节（100 MiB）。全局阈值和单文件覆盖均不能绕过该上限。

版本恢复会先确认版本元数据属于请求路径，并校验对象摘要。恢复操作复用上述流式写入事务；当前内容与目标版本不同时，会在发布前把当前文件保存为注释为 `before restore` 的安全版本。当前文件超过版本对象硬上限时，恢复在修改目标前失败。发布后的版本、索引或持久化步骤失败时，事务恢复旧内容与写入前索引快照；恢复失败则保留持久恢复证据并启用写入门禁。

保留策略示例：

```toml
[storage.retention]
max_versions = 50
max_age = "2160h"
```

```bash
  http://localhost:8080/api/v1/versions/documents/report.docx

curl -X POST \
  --config "$curl_auth_config" \
  "http://localhost:8080/api/v1/versions/abc123.../restore?path=%2Fdocuments%2Freport.docx"
```

## 与传统 NAS 对比

| 范围 | MnemoNAS | 传统 NAS | 纯 CAS 系统 |
| --- | --- | --- | --- |
| 当前文件 | 原生文件 | 原生文件 | CAS 对象 |
| 版本存储 | CAS 对象 | 文件系统快照 | CAS 对象 |
| 当前文件可读性 | 可直接读取 | 可直接读取 | 需要专用软件 |
| 去重 | BLAKE3 整对象版本；dataplane 中提供 CDC file API，但当前版本历史不会按 CDC 分块引用计数 | 依赖文件系统 | 核心功能 |
| 元数据 | SQLite | 文件系统和应用元数据 | JSON/DB |
| 复杂度 | 中 | 简单文件共享用法较低 | 高 |

混合方案牺牲一部分纯粹性，换取可恢复性和用户可检查性。
当前版本历史使用整对象 CAS 快照；FastCDC API 属于数据面能力，不表示已启用块级版本去重。

## 文件系统兼容性

MnemoNAS 不要求特定文件系统。

| 文件系统 | 兼容性 | 建议 | 说明 |
| --- | --- | --- | --- |
| ext4 | 支持 | 好 | 稳定的 Linux 默认选择 |
| XFS | 支持 | 好 | 适合大文件和并发 |
| Btrfs | 支持 | 很好 | 快照和 scrub 可增加保护层 |
| ZFS | 支持 | 最佳 | Mirror、scrub、压缩和运维模型成熟 |
| NTFS | 支持 | 有限 | 适合 Windows 环境 |
| APFS | 支持 | 好 | 适合 macOS 环境 |
| exFAT | 不建议 | 差 | 原子性预期较弱 |
| NFS mount | 谨慎支持 | 有限 | 注意延迟和一致性行为 |

## 推荐配置

| 用途 | 配置 | 数据安全 | 成本 |
| --- | --- | --- | --- |
| 预算有限 | 单盘 ext4 + 云备份 | 基础 | 低 |
| 推荐镜像 | ZFS mirror，2 块盘 | 强 | 中 |
| 进阶 Linux | Btrfs RAID1 | 强 | 中 |
| 兼容性优先 | mdadm RAID1 + ext4 | 强 | 中 |
| 大容量 | ZFS RAIDZ1 或 RAIDZ2 | 更强 | 高 |

### ZFS Mirror

```bash
sudo zpool create mnemonas mirror /dev/sda /dev/sdb
sudo zfs set mountpoint=/srv/mnemonas mnemonas
sudo zfs set compression=lz4 mnemonas
sudo zfs set recordsize=1M mnemonas
```

计划 scrub：

```cron
0 2 * * 0 /sbin/zpool scrub mnemonas
```

配置：

```toml
[storage]
root = "/srv/mnemonas"
```

### Btrfs RAID1

```bash
sudo mkfs.btrfs -m raid1 -d raid1 /dev/sda /dev/sdb
sudo mkdir -p /srv/mnemonas
sudo mount /dev/sda /srv/mnemonas
```

fstab 示例：

```bash
echo "UUID=$(blkid -s UUID -o value /dev/sda) /srv/mnemonas btrfs defaults,compress=zstd 0 0" | sudo tee -a /etc/fstab
```

Scrub：

```cron
0 2 * * 0 /sbin/btrfs scrub start /srv/mnemonas
```

### mdadm RAID1 + ext4

```bash
sudo mdadm --create /dev/md0 --level=1 --raid-devices=2 /dev/sda /dev/sdb
sudo mkfs.ext4 /dev/md0
sudo mkdir -p /srv/mnemonas
sudo mount /dev/md0 /srv/mnemonas
sudo mdadm --detail --scan | sudo tee /etc/mdadm/mdadm.conf
sudo update-initramfs -u
```

### 单盘 + 云备份

```bash
sudo mkdir -p /srv/mnemonas
```

使用 rclone、restic 或 borg 同步前，应先使用快照或冷备份窗口。单盘部署必须有离机备份。

## 数据安全层

```text
第 1 层：MnemoNAS 应用保护
  - BLAKE3 校验
  - 原子写入
  - 版本历史和回收站
  - scrub

第 2 层：文件系统保护
  - ZFS/Btrfs scrub
  - 写时复制
  - 快照

第 3 层：硬件冗余
  - mirror 或 RAID
  - 热备盘

第 4 层：独立备份
  - 云备份
  - 离线外置盘
```

| 操作 | 主要因素 |
| --- | --- |

## 总结

| 问题 | 答案 |
| --- | --- |
| 是否需要特定文件系统？ | 不需要。ext4 足以运行。 |
| 推荐什么？ | ZFS mirror 提供更强可靠性。 |
| 能满足常见 NAS 可靠性预期吗？ | 可以，但需要配合 mirror/RAID 和备份。 |
| 数据能迁移到另一台机器吗？ | 可以，迁移完整存储根目录。当前文件仍可读取；版本需要 MnemoNAS 元数据。 |

核心原则：MnemoNAS 增加应用级版本、整对象版本去重、校验和恢复能力。文件系统冗余和独立备份仍由用户负责。
